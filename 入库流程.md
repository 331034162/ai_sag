# ai_sag 文档入库流程

本文档梳理 ai_sag 知识库从**用户上传文档**到**数据持久化（MySQL + Chroma 向量库）**的完整处理链路，包含概要流程、流程细节、实现逻辑三个层次。

---

## 一、概要流程

入库链路遵循 **Loader → Cleaner → Splitter → Extractor → Persist（MySQL + 向量库）** 的经典 RAG 流水线，核心创新点在 Extractor 阶段用 LLM 抽取"事件 + 实体 + 角色"，构建 SAG 语义关联图。

```
用户上传文件/文本
       │
       ▼
┌──────────────────────────────┐
│ 1. 文档加载 (Loader)          │  按后缀分发到对应 Reader，解析为纯文本 LoadedDocument
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 2. 文本清洗 (Cleaner)         │  去标签/换行规整/空白压缩/硬断行合并
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 3. 文档切分 (Splitter)        │  按标题/句子/语义切为 Chunk 列表（默认 semantic）
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 4. 事件/实体抽取 (Extractor)  │  LLM 对每个 chunk 抽取 title/summary/content + 11类实体+角色
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 5. MySQL 事务写入             │  sources/documents/chunks/events/entities/event_entities
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 6. 向量化 + Chroma 写入       │  chunks / event_titles / event_contents / entities 四套向量
└──────────────┬───────────────┘
               │
               ▼
         标记 vector_synced=True（入库成功）
```

**核心入口**：

| 接口 | 路径 | 引擎方法 |
|------|------|----------|
| 上传文件 | `POST /api/documents` (multipart/form-data) | `IngestPipeline.ingest_file` |
| 上传纯文本 | `POST /api/documents/text` | `IngestPipeline.ingest_text` |
| 删除文档 | `DELETE /api/documents/{source_id}` | `IngestPipeline.delete_source` |
| 重建文档 | `PUT /api/documents/{source_id}` | 新入 → 成功后删旧（零空洞窗口） |

代码位置：[api.py:161-289](file:///d:/ai-llamaindex/ai_sag/api.py#L161-L289)、[ingest/pipeline.py](file:///d:/ai-llamaindex/ai_sag/ingest/pipeline.py)

### 阶段 1：文档加载（Loader）

`DocumentLoader.default()` 注册 5 个 Reader，按文件后缀分发：

| Reader | 支持后缀 | 依赖库 | 特殊处理 |
|--------|----------|--------|----------|
| `MarkdownReader` | `.md` `.markdown` | 内置 `open` | UTF-8 读取，`errors="ignore"` 容错 |
| `TextReader` | `.txt` `.text` `.log` | 内置 `open` | 同上 |
| `DocxReader` | `.docx` | `python-docx` | 遍历所有段落，按 `\n\n` 拼接 |
| `PDFReader` | `.pdf` | `PyMuPDF (fitz)` | 逐页提取文本，按页拼接 |
| `ExcelReader` | `.xlsx` | `openpyxl` + `doc_parser/excel/v2` | 解析为 Markdown 表格，含防御性校验 |

**Excel 入口的防御性校验**（[readers.py:93-125](file:///d:/ai-llamaindex/ai_sag/loader/readers.py#L93-L125)）：

1. **存在性校验**：文件不存在直接抛错；
2. **扩展名拦截**：`.xls`（旧二进制格式）要求先转 `.xlsx`；
3. **大小上限**：100MB，超过拒绝；
4. **签名校验**：读取文件头 4 字节，必须是 `PK\x03\x04`（xlsx 本质是 zip），防止后缀欺骗；
5. **样式表容错**：`doc_parser/excel/v2` 中 `_load_workbook_safe` 会在 `openpyxl` 解析 styles 失败时，替换为最小有效 `styles.xml`（预填充 1000 个空 `<xf/>`），解决非标准 Excel 报错 `expected <class 'openpyxl.styles.fills.Fill'>`。

**文件上传流程**：
- FastAPI 通过 `UploadFile.file.read()` 读取文件字节；
- 写入临时文件 `tempfile.NamedTemporaryFile(delete=False, suffix=...)`；
- 调 `loader.load(tmp_path)` → `LoadedDocument`；
- `finally` 块中 `os.unlink(tmp_path)` 清理临时文件（无论成功/失败）。

**纯文本入口**：`loader.load_text(title, content)` 直接构造 `LoadedDocument(file_type="text")`，跳过文件解析。

产物：`LoadedDocument(title, content, source_path, file_type, metadata)`

代码位置：[loader/base.py](file:///d:/ai-llamaindex/ai_sag/loader/base.py)、[loader/readers.py](file:///d:/ai-llamaindex/ai_sag/loader/readers.py)、[doc_parser/excel/v2/parser.py](file:///d:/ai-llamaindex/ai_sag/doc_parser/excel/v2/parser.py)

### 阶段 2：文本清洗（Cleaner）

`TextCleaner.clean()` 按顺序执行 5 步正则清洗（可通过构造参数开关）：

| 步骤 | 默认开 | 正则/操作 | 作用 |
|------|--------|-----------|------|
| 去 HTML 标签 | ✅ | `<[^>]+> → ""` | 清除从 Word/PDF/网页残留的标签 |
| 统一换行 | ✅ | `\r\n` / `\r → \n` | 跨平台换行统一 |
| 规整空白 | ✅ | 全角空格 `\u3000`、连续空格/制表符 → 单空格；逐行 strip | 避免 "abc　　def" 类异常 |
| 合并硬断行 | ✅ | `(?<!\n)\n(?!\n) → " "` | PDF 复制常见的"每行被硬换行"问题：单换行视为同段落内换行，替换为空格；连续空行保持段落分隔 |
| 压缩空行 | ✅ | `\n{3,} → \n\n`，整体 strip | 连续空行压缩为一个空行 |

- 清洗后返回**新的** `LoadedDocument`，`metadata["cleaned"]=True`，不修改原对象。

代码位置：[cleaner/text_cleaner.py](file:///d:/ai-llamaindex/ai_sag/cleaner/text_cleaner.py)

### 阶段 3：文档切分（Splitter）

切分器由 `create_splitter(cfg, embed_model)` 工厂根据 `cfg.splitter.mode` 创建：

| 模式 | 默认 | 说明 | 适用 |
|------|------|------|------|
| `semantic` | ✅ 默认 | `SemanticSplitterNodeParser`（percentile=95），按语义相似度断点切分 | chunk 语义最完整，**最适合 SAG 事件抽取** |
| `auto` | - | md→markdown，其余→sentence | 速度快、适配性强 |
| `markdown` | - | `MarkdownNodeParser` 按 `#` 标题切，超过 `chunk_size*4` 的长节点再用 SentenceSplitter 二级切（`_TwoStageParser`） | 原生 Markdown 文档 |
| `sentence` | - | `SentenceSplitter(chunk_size, chunk_overlap)` | docx/pdf/txt 等纯文本兜底 |
| `token` | - | `TokenTextSplitter` 按 token 精确切 | 严格 token 预算场景 |
| `code` | - | `CodeSplitter` 按语法结构 | 代码类文档 |

**默认配置**（可通过环境变量覆盖）：
- `AISAG_SPLITTER_MODE=semantic`
- `AISAG_CHUNK_SIZE=512`
- `AISAG_CHUNK_OVERLAP=100`
- `AISAG_SPLITTER_LANGUAGE=python`

**关键细节**：
- semantic 模式需要 `embed_model`，直接复用 `IngestPipeline` 已构建的 `embedder._model`（避免重复加载模型）；
- 切分前包装为 LlamaIndex `Document(text=doc.content, metadata={"title": doc.title})`；
- 每个 LlamaIndex Node 转换为 `Chunk`：
  - `id = uuid.uuid4()`；
  - `heading` 从 node.metadata 中按优先级取 `Header_1 > Header_2 > Header_3 > header > title`，都没有则 `"Introduction"`；
  - `rank_index` 按切分顺序递增（0-based）；
  - 空文本 chunk 直接丢弃。
- 切分后若 `chunks=[]`，直接抛 `ValueError("切分后无有效切片")` 终止入库。

产物：`list[Chunk]`

代码位置：[splitter/factory.py](file:///d:/ai-llamaindex/ai_sag/splitter/factory.py)、[splitter/chunk_splitter.py](file:///d:/ai-llamaindex/ai_sag/splitter/chunk_splitter.py)、[splitter/auto_splitter.py](file:///d:/ai-llamaindex/ai_sag/splitter/auto_splitter.py)

### 阶段 4：事件/实体抽取（Extractor）—— SAG 核心

`EventExtractor.extract_batch(chunks, doc_title, parallel=cfg.ingest.extract_parallel)` 把每个 chunk 送到 LLM，**抽取一个融合事件及其关联实体**。

**抽取模型**（Pydantic）：
```python
class _Entity(BaseModel):
    type: str          # 11 类实体之一
    name: str          # 归一化后的实体名
    description: str   # 实体固有属性（如"互联网银行"、"科技公司"），不随事件变化
    role: str          # 在当前事件中的角色标签（4-10字，如"贷款方"/"签约时间"/"标的物"）

class _Event(BaseModel):
    title: str
    summary: str       # ≤500 字摘要，用于快速判断相关性
    content: str       # 有机叙事主线（非要点列表）
    entities: list[_Entity]
```

**抽取 Prompt**（`extract_system_prompt()`）核心约束：

1. **强制单事件融合**：一个 chunk 只抽一个综合顶层事件，不拆成多个三元组，避免语义碎片化；
2. **全局扫描**：先识别时间/地点/主体/动作/对象/数据/因果/对比/关系，再写事件；
3. **11 类实体**（与查询侧完全一致，口径统一）：
   `time/location/person/organization/group/topic/work/product/action/metric/label`；
4. **实体命名规范**：去"有限公司/股份公司/集团"等后缀、去地名修饰、时间用阿拉伯数字，入库与查询共用同一规范；
5. **description vs role 区分**：
   - `description`：实体**固有属性**，不随事件变化（例："互联网银行"）；
   - `role`：实体**在当前事件中的角色标签**，4-10字短词（例："贷款方"、"被担保方"、"签约主体"、"标的物"、"计算依据"）；
6. **跨片段关联**：顺序抽取时，前一片段的 `summary` 作为 `previous_context` 传入下一片段，用于消解代词（他/她/它/该/其）和省略主语。

**两种抽取模式**（由 `AISAG_EXTRACT_PARALLEL` 控制）：

| 模式 | 实现 | previous_context | 速度 | 跨片段消解 |
|------|------|------------------|------|------------|
| 顺序（默认 `false`） | 单线程 for 循环 | ✅ 传递前一事件 summary | 慢 | 好 |
| 并行（`true`） | ThreadPoolExecutor(max_workers=4) + as_completed | ❌ 不传，仅用 doc_title 全局上下文 | 快 ~4x | 退化 |

**健壮性**：
- LLM 结构化抽取失败直接抛出 `ExtractionError`，不再有 `_fallback` 静默降级；
- 并行模式下任何 chunk 失败会传播异常，由上层调用方（`extract_batch`）统一处理。

产物：`list[ExtractedEvent]`（与 chunks 一一对应）

代码位置：[extractor/event_extractor.py](file:///d:/ai-llamaindex/ai_sag/extractor/event_extractor.py)、[base/prompts.py:136-178](file:///d:/ai-llamaindex/ai_sag/base/prompts.py#L136-L178)

### 阶段 5：MySQL 事务写入

`db.persist_source(...)` 在**单个事务**内完成所有结构化数据写入，保证原子性。

**写入顺序**（[mysql_store.py:198-284](file:///d:/ai-llamaindex/ai_sag/storage/mysql_store.py#L198-L284)）：

1. **前置校验**：`len(chunks) != len(events)` 直接抛异常（P0 修复，保证 1:1）；
2. `aisag_sources`：`INSERT ... ON DUPLICATE KEY UPDATE`，`metadata.vector_synced=False`（标记未完成向量同步）；
3. `aisag_documents`：插入 document 记录（存完整原文，支持后续下载/重建）；
4. `aisag_chunks`：`executemany` 批量插入所有 chunks（性能优化）；
5. `aisag_events`：`executemany` 批量插入所有 events（每个 event 绑定到对应 chunk_id）；
6. `aisag_entities` + `aisag_event_entities`（逐条处理，因为需要跨 chunk 去重）：
   - 实体去重 key：`(entity_type, normalized_name)`；
   - `_normalize(name) = re.sub(r"\s+", "", name).strip().lower()`（去空白+小写）；
   - 先在本次 `seen_entities` 字典查，再 `INSERT ... ON DUPLICATE KEY UPDATE` + `SELECT id` 查回主键（防止跨文档同名实体已存在）；
   - `aisag_event_entities.description` 存的是实体**在该事件中的 role**（供精排阶段使用）；
   - `INSERT IGNORE` 避免重复关联。

**关键设计**：
- 实体**跨 source 全局共享去重**（通过 `(entity_type, normalized_name)` 唯一索引），保证"众商贷"在 A 文档和 B 文档中是同一个实体节点，图扩展可跨文档；
- `vector_synced=False` 作为"未完成"标记，任何时刻崩溃都可通过对账任务清理；
- 所有写操作在单事务内，失败则全部回滚。

代码位置：[mysql_store.py:198-284](file:///d:/ai-llamaindex/ai_sag/storage/mysql_store.py#L198-L284)

### 阶段 6：向量化 + Chroma 写入

MySQL 事务成功后，开始向量化并写入 Chroma（共 **4 套向量**，支撑检索的不同阶段）：

| Collection | 文本来源 | 用途（检索阶段） | 是否按 source_id 过滤 |
|------------|----------|------------------|----------------------|
| `chunks` | chunk.content | 最终片段向量召回（双路融合 Chunk B + fallback） | ✅ |
| `event_titles` | event.title | 事件标题向量召回（阶段 2b） | ✅ |
| `event_contents` | event.content | 事件粗排余弦相似度（阶段 6） | ✅ |
| `entities` | entity.name | 实体模糊匹配（阶段 3） | ❌ 跨 source 共享 |

**写入顺序**：

1. **批量编码 chunks**：`embedder.aembed_texts(chunk_texts)` → `aadd_chunks`（元数据含 `source_id`）；
2. **批量编码事件标题**：`aembed_texts(title_texts)` → `aadd_events(collection="event_titles")`；
3. **批量编码事件内容**：`aembed_texts(content_texts)` → `aadd_events(collection="event_contents")`；
4. **实体名归一化后批量编码**：
   - 先用 `name_lookup` 从 ExtractedEvent 中取原始 name（避免用 normalized 丢中文语义）；
   - 去重后批量编码 → `aadd_entities`（不带 source_id，跨文档共享）；
5. 全部成功 → `db.mark_vector_synced(source_id, True)`。

所有向量写入通过 `asyncio.to_thread` 包装同步的 LlamaIndex/Chroma 调用，保持外层异步接口一致。

代码位置：[pipeline.py:102-151](file:///d:/ai-llamaindex/ai_sag/ingest/pipeline.py#L102-L151)、[chroma_store.py:27-35](file:///d:/ai-llamaindex/ai_sag/vector_store/chroma_store.py#L27-L35)

### 阶段 7：失败回滚

如果向量写入过程中任何一步失败：

```
向量写入异常
   ↓
catch vec_err
   ├─ db.delete_by_source(source_id)   → MySQL 事务删除（物理删 chunks/events/documents/sources，找出孤儿实体一并删）
   ├─ vectors.adelete_by_source        → 删 3 个事件/chunk collection 的向量
   └─ vectors.adelete_entities_by_ids  → 删本次新增的孤儿实体向量
   ↓
重新 raise，API 层返回 500
```

回滚顺序：**MySQL 先删 → 向量库后删**；MySQL 删完会返回 `orphan_entity_ids`（那些只在该 source 中出现、删完不再被任何事件引用的实体），一并清理向量。

代码位置：[pipeline.py:152-162](file:///d:/ai-llamaindex/ai_sag/ingest/pipeline.py#L152-L162)

### 阶段 8：后台对账（Reconcile）

为防止任何阶段崩溃导致 MySQL 与向量库不一致，后台每隔 `reconcile_interval=300s` 执行一次对账，清理三类不一致数据：

| 类型 | 现象 | 清理动作 |
|------|------|----------|
| ①未同步 source | MySQL 中 `vector_synced=0` 的 source（入库崩溃、删除中崩溃） | 删 MySQL → 删向量库 |
| ②孤儿 source 向量 | 向量库有但 MySQL 无此 source_id（删除流程中物理删 source 后向量删失败） | 向量库按 source_id 删 |
| ③孤儿实体向量 | entities 向量有但 MySQL entities 表无此 id（删除步骤4失败残留） | 向量库按 entity_id 批量删 |

对账启动时立即执行一次，之后每 `reconcile_interval` 秒执行一次；任何一类清理失败不影响其它类，日志 warning 后下次重试。设 `AISAG_RECONCILE_INTERVAL=0` 可禁用定时对账（仅启动时对一次）。

代码位置：[pipeline.py:208-270](file:///d:/ai-llamaindex/ai_sag/ingest/pipeline.py#L208-L270)

## 三、实现逻辑要点

### 3.1 核心设计决策

1. **统一 11 类实体口径（入库 = 查询）**
   - `ENTITY_TYPES / ENTITY_NAME_NORM / entity_types_detail()` 都定义在 `base/prompts.py`；
   - 查询侧 `query_extract_system_prompt()` 复用同一套规范，从根本上避免"入库抽的是'华为技术有限公司'、查询抽的是'华为'"导致对不上的问题。

2. **单事件融合原则**
   - 每个 chunk 只抽一个综合顶层事件，不拆三元组；
   - 事件 `content` 是有机叙事主线而非要点列表，后续作为 `event_contents` 向量文本语义完整。

3. **description vs role 双字段设计**
   - `entities.description`：实体**固有属性**，全局不变（向量实体节点的静态属性）；
   - `event_entities.description`（关联表）：实体**在该事件中的角色**，随事件变化；
   - 查询精排阶段把"众商贷(贷款方)"这类角色信息渲染给 LLM，解决方向性问题（"谁贷给谁"）。

4. **实体跨 source 全局去重共享**
   - 唯一键 `(entity_type, normalized_name)`，归一化规则：去空白 + 小写；
   - 向量实体 collection 不按 source_id 过滤（任何文档中的"众商贷"都是同一个向量节点）；
   - 删文档时必须检查实体是否还被其它事件引用，只删真正的孤儿。

5. **chunks 与 events 严格 1:1**
   - 每个 chunk 对应一个事件，每个事件绑定一个 chunk_id；
   - persist_source 入口强校验，避免长度不匹配导致错位；
   - LLM 抽取失败直接抛出 `ExtractionError`，上层负责重试或记录。

6. **写入顺序：先 MySQL 后向量，失败双向回滚**
   - MySQL 先写（带 `vector_synced=False` 标记）→ 再写向量 → 最后标记 True；
   - 任意一步失败都清理两边残留，结合定时对账兜底，最终一致。

7. **删除顺序：先向量后 MySQL（零空洞反向设计）**
   - 标记 `vector_synced=False` → 先删向量库（检索立即查不到）→ 再删 MySQL（页面稍后消失）；
   - 避免"页面显示已删除但检索还能搜到"的尴尬窗口。

8. **重建策略：先入新后删旧**
   - `PUT /api/documents/{source_id}` 接口用新 source_id 入库成功后再删旧 source_id；
   - 入库失败旧数据完全不受影响，避免"先删后入失败导致文档消失"。

9. **semantic 切分为默认**
   - semantic 切出的 chunk 按语义断点划分，天然保持语义完整，LLM 抽取事件质量最高；
   - 代价：入库时需要额外的 embedding 计算（复用已有 embedder），速度比 sentence 慢，但为一次性成本。

10. **Excel 解析容错（v2 版本）**
    - 四层防御校验（存在/扩展名/大小/ZIP 签名）；
    - `_load_workbook_safe` 对样式表损坏的非标准 xlsx，替换为最小有效 styles.xml（预填充 1000 个 `<xf/>`），覆盖 99% 实际损坏场景。

### 3.3 关键算法与工具方法

**实体归一化**（`MysqlStore._normalize`）：
```python
re.sub(r"\s+", "", name or "").strip().lower()
```
注意：仅用于去重 key；向量编码和展示用原始 name 保留中文语义。

**Excel 样式表兜底**（`_load_workbook_safe`）：
- 构造包含 1000 个 `<xf/>` 占位的最小 styles.xml；
- 用 `zipfile` 拷贝原 xlsx 内容到临时文件，替换 `xl/styles.xml`；
- 加载临时文件（`_MAX_STYLE_ID_FALLBACK=1000` 覆盖绝大多数实际样式 id）。

**两级切分器**（`_TwoStageParser`）：
- 先用 MarkdownNodeParser 按标题切；
- 单节点超过 `chunk_size*4`（默认 2048 字符）再用 SentenceSplitter 二级切；
- metadata 合并，避免标题信息丢失。

### 3.4 异常处理与可观测性

- **日志打点**：每个关键节点打 INFO 日志（"入库开始"/"切分完成 chunk数=N"/"事件抽取完成 事件数=N 实体总数=M"/"MySQL 写入完成"/"入库完成"）；
- **异常分类**：
  - 文件校验失败 → HTTP 400（用户输入问题，不回滚）；
  - LLM 抽取单 chunk 失败 → 抛出 `ExtractionError`，由 `extract_batch` 统一处理；
  - MySQL 事务失败 → 自然回滚，异常向上抛；
  - 向量写入失败 → 主动回滚 MySQL + 清理向量，raise；
  - 定时对账单 source 失败 → catch 后继续处理下一个；
- **trace/metadata**：每个 chunk 带 `metadata.node_id`（LlamaIndex 节点 id）、`cleaned=True`；source 的 `metadata.vector_synced` 是一致性标记。

## 五、入库与问答的数据对接

入库阶段产出的数据如何被问答链路使用：

| 入库产物 | 问答环节 | 作用 |
|----------|----------|------|
| `chunks` + `chunks` 向量 | 双路融合 Chunk B + fallback | 纯向量兜底召回原文片段 |

| `events` ↔ `chunks` 绑定 | 切片回取（阶段 8） | 精排后的事件直接回取对应 chunk 原文 |
| `event_entities` 图关联 | 多跳 BFS 扩展（阶段 5） | 沿"事件↔实体↔事件"边跨事件推理 |
