# 内容字段抽取逻辑与作用详解

> 本文档梳理 `aisag_events`（title / summary / content）、`aisag_chunks`（content）、`aisag_documents`（content）三类核心内容字段的抽取逻辑、抽取后大小，以及在问答流程中发挥的作用。

---

## 一、字段总览

| 表 | 字段 | 类型 | 约束 | 抽取者 | 是否入向量库 |
|----|------|------|------|--------|-------------|
| `aisag_documents` | `content` | MEDIUMTEXT | **可空** | 文件加载器 | ❌ 不入 |
| `aisag_chunks` | `content` | MEDIUMTEXT | **NOT NULL** | 切分器 | ✅ `chunks` 集合 |
| `aisag_events` | `title` | VARCHAR(1024) | NOT NULL | LLM 抽取 | ✅ `event_titles` 集合 |
| `aisag_events` | `summary` | TEXT | 可空 | LLM 抽取 | ❌ 不入 |
| `aisag_events` | `content` | MEDIUMTEXT | NOT NULL | LLM 抽取 | ✅ `event_contents` 集合 |

**关键认知**：`events.content` ≠ `chunks.content`，前者是 LLM 重新组织的精炼叙事，后者是切分器产出的原文片段。

---

## 二、抽取逻辑详解

### 2.1 `aisag_documents.content` —— 文档完整原文

#### 抽取流程

```
用户上传文件
    ↓
LoadedDocument（加载器读取全文）
    ↓
db.insert_document(content=LoadedDocument.content)
    ↓
aisag_documents.content
```

#### 抽取者

- **文件加载器**（`doc_parser/` 下的 pdf/docx/md/xlsx 解析器）
- 无 LLM 参与，纯文本提取

#### 抽取规则

- 保留文件完整原文（含格式、表格、标题层级）
- markdown 文件原样存储
- pdf/docx 转为纯文本后存储

#### 大小

- 与原始文件大小相当
- **可空**：URL/流式入库时可仅保留元数据，content 为空

---

### 2.2 `aisag_chunks.content` —— 切片原文

#### 抽取流程

```
aisag_documents.content（完整原文）
    ↓
ChunkSplitter.split_text()
    ↓
按 chunk_size=512, chunk_overlap=100 切分
    ↓
aisag_chunks.content（每片 ~512 字符）
```

#### 抽取者

- **切分器**（`splitter/chunk_splitter.py`）
- 无 LLM 参与，纯规则切分

#### 切分模式

| 模式 | 切分器 | 适用场景 |
|------|--------|---------|
| `markdown`（默认） | `MarkdownNodeParser` + `SentenceSplitter` 兜底 | .md 文件 |
| `sentence` | `SentenceSplitter(chunk_size=512, chunk_overlap=100)` | docx/pdf/txt |
| `token` | `TokenTextSplitter` | 严格 token 预算场景 |
| `code` | `CodeSplitter` | 代码类文档 |

#### 切分参数（默认值）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `AISAG_CHUNK_SIZE` | `512` | 单片字符数 |
| `AISAG_CHUNK_OVERLAP` | `100` | 片段间重叠字符数 |
| `AISAG_SPLITTER_MODE` | `markdown` | 切分模式 |

#### 大小

- **单片段**：约 400-512 字符（受重叠和句子边界影响）
- **总片段数**：原文长度 / (chunk_size - chunk_overlap)
- 50KB 文档约产生 10 个 chunks

#### 特点

- 保留原文格式（标题、列表、表格内容）
- 是**原文的物理切片**，无语义加工
- 必填（NOT NULL）

---

### 2.3 `aisag_events.title` —— 事件标题

#### 抽取流程

```
aisag_chunks.content（切片原文）
    ↓
EventExtractor.extract() 调用 LLM.structured_predict
    ↓
LLM 按 EXTRACT_TEMPLATE 抽取
    ↓
aisag_events.title（~15 字短标题）
```

#### 抽取者

- **LLM**（`extractor/event_extractor.py`）
- 通过 `structured_predict` 返回结构化结果

#### Prompt 指令

> "全局扫描：先识别时间、地点、主体、动作、对象、数据、评价、因果、对比、关系，再写事件。"
> "强制单事件：把片段中所有有效信息融合为一个综合顶层事件。"

#### 大小

- **约 10-25 字**
- VARCHAR(1024) 容量足够

#### 特点

- 概括本片段的核心事件主题
- 短小精炼，用于快速匹配

---

### 2.4 `aisag_events.summary` —— 事件摘要

#### 抽取流程

同 title，由 LLM 在同一次 `structured_predict` 调用中一并生成。

#### Prompt 指令

> "summary 应为不超过500字的压缩摘要，用于快速判断事件相关性。"

#### 大小

- **≤ 500 字**（Prompt 强制约束）
- TEXT 类型，可承载更长但实际较短

#### 特点

- 压缩的事件主旨
- 用于 LLM 精排时快速判断相关性
- **不入向量库**

---

### 2.5 `aisag_events.content` —— 事件叙事主线

#### 抽取流程

```
aisag_chunks.content（切片原文）
    ↓
LLM.structured_predict(_ExtractionResult)
    ↓  ⚠️ LLM 重新组织，非复制原文
aisag_events.content（精炼叙事主线）
```

#### 抽取者

- **LLM**（`extractor/event_extractor.py` 的 `extract` 方法）
- 通过 `structured_predict` 返回 `_Event.content`

#### Prompt 指令

> "忠实性：不杜撰、不遗漏核心事实、不改主语、不长段照抄原文。"
> "content 应为有机叙事主线，非要点列表。"

**关键约束**：
- "不长段照抄原文" → LLM 不会复制 chunk.content
- "有机叙事主线" → LLM 写成精炼连贯的叙述
- "非要点列表" → 不是列表，是叙述性文本

#### 容错

LLM 结构化抽取失败时直接抛出 `ExtractionError`，不再静默降级。上层调用方（如 `extract_batch`）负责重试或记录异常。**无** Fallback 路径。

#### 大小

- **约 100-200 字**（比 chunk.content 短一半以上）
- MEDIUMTEXT NOT NULL

#### 特点

- **LLM 重新组织的精炼叙事**，非原文复制
- 短、连贯、去格式、整合性强
- 与 `chunks.content` **内容不同、语义不同**

## 三、抽取后大小对比

### 3.1 单字段大小对比

| 字段 | 抽取者 | 典型字数 | 相对大小 |
|------|--------|---------|---------|
| `documents.content` | 加载器 | 文档全文（~50KB） | 100% |
| `chunks.content` | 切分器 | 400-512 字符/片 | 100%（基准） |
| `events.title` | LLM | 10-25 字 | 5% |
| `events.summary` | LLM | ≤ 500 字 | 15% |
| `events.content` | LLM | 100-200 字 | 30-40% |

### 3.2 单文档存储估算（50KB 文档，10 个 chunks）

| 存储位置 | 字段 | 估算大小 |
|---------|------|---------|
| MySQL `aisag_documents` | content | 50 KB |
| MySQL `aisag_chunks` | content × 10 | 50 KB |
| MySQL `aisag_events` | title × 10 | 0.5 KB |
| MySQL `aisag_events` | summary × 10 | 2 KB |
| MySQL `aisag_events` | content × 10 | 20 KB |
| Chroma `chunks` | text + vec × 10 | 70 KB |
| Chroma `event_titles` | text + vec × 10 | 21 KB |
| Chroma `event_contents` | text + vec × 10 | 40 KB |
| Chroma `entities` | text + vec × 20 | 40 KB |
| **总计** | — | **~293 KB** |
| **膨胀倍数** | — | **约 6 倍** |

## 四、在问答流程中的作用

### 4.1 问答流程 9 环节总览

```
1. 查询重写        [LLM]
2. 实体抽取        [LLM]
3. 实体召回        [向量库] entities
4. 标题召回        [向量库] event_titles
5. 多跳扩展        [数据库] aisag_event_entities
6. 事件粗排        [向量库] event_contents
7. LLM 精排        [LLM] events.summary
8. 切片回取        [数据库] aisag_chunks
9. 双路融合+生成   [LLM] chunks.content
```

### 4.2 各字段参与环节详表

| 字段 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 | 9 |
|------|---|---|---|---|---|---|---|---|---|
| `documents.content` | — | — | — | — | — | — | — | — | — |
| `chunks.content` | — | — | — | — | — | — | — | ✅ | ✅ |
| `events.title` | — | — | — | ✅ | — | — | — | — | — |
| `events.summary` | — | — | — | — | — | — | ✅ | — | — |
| `events.content` | — | — | — | — | — | ✅ | — | — | — |

**说明**：✅ 表示该环节直接使用此字段。

### 4.3 各字段具体作用

#### `aisag_documents.content` —— ❌ 不参与问答流程

| 作用点 | 描述 |
|--------|------|
| 文档详情 API | `GET /api/documents/{id}` 返回完整原文展示 |
| 文档下载 API | `GET /api/documents/{id}/download` 流式输出 .md 文件 |
| 全文检索 API | `GET /api/documents/search?keyword=xxx` 用 LIKE 匹配 |

**不参与任何检索/生成环节**，是用户级的"原文备份"。

---

#### `aisag_chunks.content` —— ✅ 检索核心，参与 2 个环节

##### 环节 8：切片回取（**最终片段来源**）

```
精排后 events → event.chunk_id → 回查 aisag_chunks → 用 chunk.content 作为最终片段
```

- 精排选出 TopK 事件后，通过 `event.chunk_id` 反查 `aisag_chunks` 表
- **用 `chunk.content` 作为最终返回给 LLM 的片段原文**
- 是问答系统喂给 LLM 的"命脉"

##### 环节 9：双路融合 Path B（向量召回路）

```
query 向量 vs chunks 向量 → 召回 chunks → chunk.content 作为 Path B 片段
```

- Path B 不走事件路，直接用 query 向量在 `chunks` collection 召回
- 用 `chunk.content` 作为返回片段
- 与 Path A 事件路融合，互补召回

---

#### `aisag_events.title` —— ✅ 参与环节 4

##### 环节 4：标题召回

```
query 向量 vs event_titles 向量 → 召回 Top N 事件
```

- 入库时 `events.title` 编码成向量存入 Chroma `event_titles` collection
- 检索时用 query 向量召回标题相关的事件
- 作为 Path A 事件路的**召回入口**之一

---

#### `aisag_events.summary` —— ✅ 参与环节 7

##### 环节 7：LLM 精排

```
粗排 Top N 事件 → events.summary 喂 LLM → LLM 打分排序
```

- 粗排后取 Top N 事件，把 `events.title` + `events.summary` + `events.entity_roles` 作为候选描述送入 LLM
- LLM 根据title/summary及实体角色（如"华为(贷款方)""2024年(签约时间)"）判断与 query 的相关性，输出重排分数
- **用 summary 而非 content**：因为 summary 精炼（≤500字），加上 title 和 roles 信息足够判断相关性

---

#### `aisag_events.content` —— ✅ 参与环节 6

##### 环节 6：事件粗排

```
候选事件 IDs → 从 Chroma event_contents 取向量 → query_vec 余弦相似度 → 排序
```

- 入库时 `events.content` 编码成向量存入 Chroma `event_contents` collection
- 检索时从 `event_contents` 取这些向量，与 query 向量算余弦相似度
- 按相似度排序，作为粗排结果

**关键澄清**：
- 粗排用的是 `events.content` 编码的**向量**，不读原文
- `events.content` 原文只在入库时编码一次，检索时不读
- 最终返回 LLM 的是 `chunks.content`，不是 `events.content`

---

## 五、双语义层设计

### 5.1 设计意图

```
原文层（chunks）
  ├─ content = 切分器原文（512 字符）
  ├─ 向量 = "原文级"语义
  └─ 用途：Path B 召回、最终回取喂 LLM
                ↓ 不同语义
事件层（events）
  ├─ content = LLM 精炼叙事（100-200 字）
  ├─ 向量 = "事件级"语义
  └─ 用途：Path A 粗排
```

### 5.2 两层向量代表的语义

| 向量来源 | 语义类型 | 检索特性 |
|---------|---------|---------|
| `chunks` 向量 | 原文细节语义 | 召回原文细节匹配的片段 |
| `event_contents` 向量 | 事件主旨语义 | 召回事件主题相关的片段 |

### 5.3 双路融合的真正价值

| 路 | 用什么编码 | 召回特点 |
|----|----------|---------|
| Path A 事件路 | `events.content` 向量 | 主题/事件级召回 |
| Path B 向量路 | `chunks.content` 向量 | 原文/细节级召回 |

**两路用不同语义的向量，互补召回**，这是双路融合的真正设计意图——不是冗余，是**两层语义互补**。

## 六、数据流全景图

### 6.1 入库阶段

```
原始文档
    │
    ├─→ documents.content    (加载器，完整原文)
    │       └─ 用于：文档详情、下载、全文检索（不参与问答）
    │
    ├─→ chunks.content       (切分器，~512 字/片)
    │       ├─ 用于：向量编码 → Chroma chunks 集合
    │       ├─ 用于：精排后回取原文
    │       └─ 用于：Path B 向量召回
    │
    └─→ events.title/summary/content   (LLM 抽取)
            ├─ title   → 向量编码 → Chroma event_titles
            ├─ summary → 仅 MySQL（精排用）
            └─ content → 向量编码 → Chroma event_contents
```

### 6.2 检索阶段

```
用户 Query
    ↓ 查询重写
eff_query
    ↓ 实体抽取
query_entities + query_vec
    │
    ├─ Path A 事件路 ─────────────────────────────┐
    │   ├─ 环节 3 实体召回    [向量库] entities   │
    │   ├─ 环节 4 标题召回    [向量库] event_titles   ← 用 events.title 向量
    │   ├─ 环节 5 多跳扩展    [数据库]               │
    │   ├─ 环节 6 事件粗排    [向量库] event_contents ← 用 events.content 向量
    │   ├─ 环节 7 LLM 精排    [LLM]                  ← 用 events.summary 原文
    │   └─ 环节 8 切片回取    [数据库]               ← 用 chunks.content 原文
    │                                                  ↓
    ├─ Path B 向量路 ─────────────────────────────┐   │
    │   └─ query_vec vs chunks 向量 → 召回        │   │
    │       → chunk.content 作为片段              │   │
    └──────────────────────────────────────────────┘   │
                        ↓ 融合                          │
              最终 sections（来自 chunks.content） ────┘
                        ↓
                 环节 9 LLM 生成答案
```

## 七、核心结论

### 7.1 各字段定位

| 字段 | 定位 | 是否可删 |
|------|------|---------|
| `documents.content` | 用户级备份（下载/展示） | ⚠️ 可空，删后影响下载 |
| `chunks.content` | **检索核心**（喂 LLM 的原文） | ❌ 删后问答瘫痪 |
| `events.title` | 标题召回入口 | ❌ 删后环节 4 失效 |
| `events.summary` | LLM 精排候选描述 | ❌ 删后环节 7 失效 |
| `events.content` | 事件级粗排向量源 | ❌ 删后丢失事件级语义 |

### 7.2 关键认知修正

| 错误认知 | 正确事实 |
|---------|---------|
| events.content = chunks.content | ❌ events.content 是 LLM 精炼的叙事主线 |
| events.content 是冗余设计 | ❌ 是双语义层设计 |
| 双路融合是冗余 | ❌ 是两层语义互补 |
| 可以删除 events.content | ❌ 会丢失事件级语义召回能力 |

### 7.3 一句话总结

> **`documents.content` 是"备份"，`chunks.content` 是"检索核心"，`events.title/summary/content` 是"事件路三级语义"**——五者各司其职，共同构成 ai_sag 的双语义层检索体系，缺一不可。
