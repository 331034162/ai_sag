# 实体字段抽取逻辑与作用详解

> 本文档梳理 `aisag_entities`（name / normalized_name / description）和 `aisag_event_entities`（description）字段的抽取逻辑、抽取后大小，以及在问答流程中发挥的作用。

---

## 一、字段总览

### 1.1 `aisag_entities` 表

| 字段 | 类型 | 约束 | 抽取者 | 是否入向量库 |
|------|------|------|--------|-------------|
| `name` | VARCHAR(512) | NOT NULL | LLM 抽取 | ✅ `entities` 集合（编码向量） |
| `normalized_name` | VARCHAR(512) | NOT NULL | 代码归一化 | ❌ 仅 MySQL（去重键） |
| `description` | TEXT | 可空 | LLM 抽取 | ❌ 仅 MySQL |

**唯一键**：`UNIQUE KEY uq_aisag_entity (entity_type, normalized_name)` —— 实体跨文档全局共享去重。

### 1.2 `aisag_event_entities` 表（事件↔实体 M:N 关联表）

| 字段 | 类型 | 约束 | 抽取者 | 实际存储内容 |
|------|------|------|--------|------------|
| `event_id` | VARCHAR(36) | NOT NULL | 代码关联 | 事件 ID |
| `entity_id` | VARCHAR(36) | NOT NULL | 代码关联 | 实体 ID |
| `weight` | FLOAT | DEFAULT 1.0 | 代码固定 | 固定 1.0 |
| `description` | TEXT | 可空 | LLM 抽取 | **实际存的是 `entity.role`（角色标签）** |

**关键认知**：`aisag_event_entities.description` 字段名是"描述"，但**实际存储的是 LLM 抽取的 `entity.role`（角色标签）**，而非 `entity.description`。

## 二、抽取逻辑详解

### 2.1 `aisag_entities.name` —— 实体原始名称

#### 抽取流程

```
aisag_chunks.content（切片原文）
    ↓
EventExtractor.extract() 调用 LLM.structured_predict
    ↓
LLM 返回 _Entity(name="华为", type="organization", description="...", role="贷款方")
    ↓
aisag_entities.name = "华为"
```

#### 抽取者

- **LLM**（`extractor/event_extractor.py`）
- 通过 `structured_predict` 返回 `_Entity.name`

#### Prompt 约束

> "实体仅作为索引和扩展点，不承载完整语义。"
> "并列实体（如'A 和 B'）拆开。"

#### 大小

- **约 2-30 字**（公司名/人名/地名等）
- VARCHAR(512) 容量足够

#### 特点

- 保留实体原始名称（含大小写、空格）
- 用于精排时展示给 LLM（如 "华为(贷款方)"）

---

### 2.2 `aisag_entities.normalized_name` —— 归一化名称（去重键）

#### 抽取流程

```
LLM 抽取 entity.name = "华为 技术 有限公司"
    ↓
MysqlStore._normalize(name)
    ↓
re.sub(r"\s+", "", name).strip().lower()
    ↓
normalized_name = "华为技术有限公司"
```

#### 抽取者

- **代码归一化**（`storage/mysql_store.py:584`）

```python
@staticmethod
def _normalize(name: str) -> str:
    return re.sub(r"\s+", "", name or "").strip().lower()
```

#### 归一化规则

| 步骤 | 操作 | 示例 |
|------|------|------|
| 1 | 去除所有空白字符 | `"华为 技术 有限公司"` → `"华为技术有限公司"` |
| 2 | 去首尾空格 | `" 华为 "` → `"华为"` |
| 3 | 转小写 | `"Huawei"` → `"huawei"` |

#### 大小

- 与 `name` 相同，约 2-30 字
- VARCHAR(512)

#### 特点

- **跨文档全局去重的核心**
- 通过 `UNIQUE KEY (entity_type, normalized_name)` 保证"众商贷"在 A 文档和 B 文档中是同一个实体节点
- 是图扩展可跨文档的基础

---

### 2.3 `aisag_entities.description` —— 实体固有属性

#### 抽取流程

```
LLM 抽取 _Entity(description="互联网银行")
    ↓
db.upsert_entity(entity)
    ↓
aisag_entities.description = "互联网银行"
```

#### 抽取者

- **LLM**（`extractor/event_extractor.py`）
- 通过 `structured_predict` 返回 `_Entity.description`

#### Prompt 约束

> "每个 entity.description 必须是该实体的**固有属性**（如'互联网银行'、'科技公司'、'某公司CEO'），**不随事件变化**，建议不超过20字。"

#### 大小

- **≤ 20 字**（Prompt 约束）
- TEXT 类型，实际很短

#### 特点

- 描述实体的**固有属性**，不随事件变化
- 与 `entity.role`（角色，随事件变化）严格区分
- ⚠️ **当前检索流程实际未读取此字段**（见第四章分析）

---

### 2.4 `aisag_event_entities.description` —— **实际存储的是角色（role）**

#### 抽取流程

```
LLM 抽取 _Entity(role="贷款方")
    ↓
db.persist_batch() 写入关联表
    ↓
INSERT INTO aisag_event_entities (..., description) VALUES (..., e.role)
                                                                  ↑
                                                          ⚠️ 传的是 role，不是 description
```

#### 代码证据

[storage/mysql_store.py:277-281](file:///d:/ai-llamaindex/ai_sag/storage/mysql_store.py#L277-L281)

```python
await cur.execute(
    "INSERT IGNORE INTO aisag_event_entities (id, event_id, entity_id, weight, description) "
    "VALUES (%s,%s,%s,%s,%s)",
    (str(uuid.uuid4()), event.id, eid, 1.0, e.role),    # ← 第 5 个参数传的是 e.role
)
```

**关键事实**：字段名是 `description`，但**实际写入的是 `entity.role`**。

#### 抽取者

- **LLM**（`extractor/event_extractor.py`）
- LLM 返回 `_Entity.role`，被存入 `event_entities.description` 字段

#### Prompt 约束

> "每个 entity.role 必须是该实体在当前事件中扮演的**角色标签**，4-10字短词，不要写成描述性长句。"
> - organization/person: 贷款方/担保方/被投资方/签约主体/被监管方
> - location: 事件发生地/注册地/约定履行地
> - time: 签约时间/生效时间/违约起始日
> - product: 标的物/被担保物/付款对象
> - metric: 被引用指标/计算依据/限定条件
> - action: 主语行为/约定行为
> - group: 受益方/被约束方
> - topic/work/label: 主题/被引作品/分类标签

#### 大小

- **4-10 字**（Prompt 约束）
- TEXT 类型，实际很短

#### 特点

- **字段名误导**：名为 `description`，实为 `role`
- 描述实体在**特定事件中**的角色（随事件变化）
- 与 `entities.description`（固有属性，不随事件变化）严格区分
- ✅ **在精排环节被读取使用**（见第四章）

## 三、抽取后大小对比

### 3.1 单字段大小对比

| 表.字段 | 抽取者 | 典型字数 | 实际内容 |
|---------|--------|---------|---------|
| `entities.name` | LLM | 2-30 字 | 实体原始名称 |
| `entities.normalized_name` | 代码归一化 | 2-30 字 | 去空格+小写后的名称 |
| `entities.description` | LLM | ≤ 20 字 | 实体固有属性 |
| `event_entities.description` | LLM | 4-10 字 | **实际是角色（role）** |
| `event_entities.weight` | 代码固定 | 1 个 float | 固定 1.0 |

### 3.2 单文档存储估算（50KB 文档，10 个 chunks，每 chunk 抽 5 个实体）

| 存储位置 | 字段 | 估算大小 |
|---------|------|---------|
| MySQL `aisag_entities` | name × 50 | 1 KB |
| MySQL `aisag_entities` | normalized_name × 50 | 1 KB |
| MySQL `aisag_entities` | description × 50 | 1 KB |
| MySQL `aisag_event_entities` | description(role) × 50 | 0.5 KB |
| MySQL `aisag_event_entities` | weight × 50 | 0.2 KB |
| Chroma `entities` | text + vec × 50 | 105 KB（向量占大头） |
| **总计** | — | **~108 KB** |

**实体相关存储占比小**，主要开销在 Chroma 的向量存储。

## 四、在问答流程中的作用

### 4.1 问答流程 9 环节总览

```
1. 查询重写        [LLM]
2. 实体抽取        [LLM]              ← 从 query 抽取实体名
3. 实体召回        [向量库+数据库]     ← 用 entities.name 查询
4. 标题召回        [向量库] event_titles
5. 多跳扩展        [数据库]            ← 用 event_entities 关联表
6. 事件粗排        [向量库] event_contents
7. LLM 精排        [LLM]               ← 用 event_entities.description(role)
8. 切片回取        [数据库] aisag_chunks
9. 双路融合+生成   [LLM]
```

### 4.2 各字段参与环节详表

| 字段 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 | 9 |
|------|---|---|---|---|---|---|---|---|---|
| `entities.name` | — | — | ✅ | — | — | — | ✅ | — | — |
| `entities.normalized_name` | — | — | ✅ | — | — | — | — | — | — |
| `entities.description` | — | — | — | — | — | — | — | — | — |
| `event_entities.description`(role) | — | — | — | — | ✅ | — | ✅ | — | — |
| `event_entities.weight` | — | — | — | — | — | — | — | — | — |

**说明**：✅ 表示该环节直接使用此字段。

### 4.3 各字段具体作用

#### `aisag_entities.name` —— ✅ 参与环节 3 和 7

##### 环节 3：实体召回（向量编码源）

```
LLM 从 query 抽取实体名 → 编码向量 → Chroma entities 召回
```

- 入库时 `entities.name` 编码成向量存入 Chroma `entities` collection
- 检索时用 query 抽取的实体名编码向量，在 `entities` 召回 Top 5 相似实体

##### 环节 7：LLM 精排（角色展示）

```
精排时调用 db.get_entity_names_by_ids(query_entity_ids) 取实体名
→ 格式化为 "华为(贷款方)" 喂 LLM
```

- 精排时通过 `get_entity_names_by_ids` 查询实体名
- 在 `_format_entity_roles` 中格式化为 `"华为(贷款方)"`、`"2024年(签约时间)"`
- 帮助 LLM 理解事件与查询实体的关联

---

#### `aisag_entities.normalized_name` —— ✅ 参与环节 3

##### 环节 3：实体召回（精确匹配）

```
LLM 从 query 抽取实体名 → _normalize 归一化 → 精确匹配 normalized_name
```

- 检索时把 query 抽取的实体名归一化
- 在 `search_entities_by_name` 中用 `normalized_name IN (...)` 精确匹配
- 是实体召回的**双路之一**（另一路是向量召回）

##### 跨文档去重（入库时）

```
INSERT INTO aisag_entities ... ON DUPLICATE KEY UPDATE
UNIQUE KEY (entity_type, normalized_name)
```

- 入库时通过 `(entity_type, normalized_name)` 唯一键去重
- 保证"众商贷"在 A 文档和 B 文档中是同一个实体节点
- 是多跳扩展可跨文档的基础

---

#### `aisag_entities.description` —— ❌ 当前流程未使用

##### 设计意图（未实现）

- 描述实体固有属性（如"互联网银行"）
- 本可用于实体消歧或 LLM 精排补充信息

##### 实际情况

- `search_entities_by_name` 查询了 `en.description`，但返回后仅取 `id`，未使用 description
- **当前检索流程实际未读取此字段**
- ⚠️ 属于"设计了但未充分利用"的字段

---

#### `aisag_event_entities.description`（实际是 role）—— ✅ 参与环节 5 和 7

##### 环节 5：多跳扩展（关联查询）

```
seed_events → 查 event_entities → 取 entity_ids
→ 查 event_entities → 取新 event_ids
→ BFS 多跳扩展
```

- 通过 `event_entities` 关联表做 BFS 扩展
- 注意：多跳扩展只用 `event_id` 和 `entity_id`，**不读 description**
- description 在多跳阶段只是"附带字段"，未被使用

##### 环节 7：LLM 精排（**核心作用**）

```
get_events_by_ids → _entity_roles_of_events
→ 读取 event_entities.description 作为 role
→ 格式化为 "华为(贷款方)" 喂 LLM 精排
```

- 精排时通过 `_entity_roles_of_events` 读取 `event_entities.description`
- 作为 `entity_roles` 映射的 value（即 role）
- 在 `_format_entity_roles` 中格式化为 `"华为(贷款方)"`、`"2024年(签约时间)"`
- 帮助 LLM 理解每个事件中实体扮演的角色，判断相关性

**这是此字段的核心作用**。

## 六、数据流全景图

### 6.1 入库阶段

```
aisag_chunks.content（切片原文）
    ↓
LLM.structured_predict(_ExtractionResult)
    ↓
_Entity(name, type, description, role)
    │
    ├─→ aisag_entities.name             (原始名称)
    │       └─ 用于：精排展示 "华为(贷款方)"
    │
    ├─→ aisag_entities.normalized_name   (代码归一化)
    │       ├─ 用于：跨文档去重唯一键
    │       └─ 用于：实体召回精确匹配
    │
    ├─→ aisag_entities.description      (LLM 抽取的固有属性)
    │       └─ ⚠️ 当前流程未使用
    │
    ├─→ Chroma entities.text = name     (向量编码源)
    │       └─ 用于：实体向量召回
    │
    └─→ aisag_event_entities.description = role   (⚠️ 字段名误导)
            └─ 用于：精排时格式化 "华为(贷款方)"
```

### 6.2 检索阶段

```
用户 Query
    ↓ 查询重写 → 实体抽取
query_entities (实体名列表)
    │
    ├─ 环节 3 实体召回 ──────────────────────────┐
    │   ├─ 精确匹配：name → _normalize →           │
    │   │              normalized_name IN (...)     │  ← 用 normalized_name
    │   └─ 向量召回：name → 编码 →                  │
    │              Chroma entities 召回             │  ← 用 name 向量
    └──────────────────────────────────────────────┘
                    ↓ entity_ids
    ┌─ 环节 5 多跳扩展 ──────────────────────────┐
    │  seed_events → event_entities → entity_ids  │
    │  → event_entities → new_event_ids           │  ← 用关联表（不读 description）
    │  → BFS 扩展                                 │
    └──────────────────────────────────────────────┘
                    ↓ event_ids
    ┌─ 环节 7 LLM 精排 ──────────────────────────┐
    │  get_events_by_ids → _entity_roles_of_events│
    │  → 读取 event_entities.description(role)    │  ← 用 event_entities.description
    │  → _format_entity_roles                     │
    │  → "华为(贷款方)、2024年(签约时间)"          │
    │  → 喂 LLM 精排                              │
    └──────────────────────────────────────────────┘
```

## 七、核心结论

### 7.1 各字段定位

| 字段 | 定位 | 是否可删 |
|------|------|---------|
| `entities.name` | 实体标识，向量编码源，精排展示 | ❌ 删后实体召回失效 |
| `entities.normalized_name` | 跨文档去重键，精确匹配 | ❌ 删后实体无法去重和精确匹配 |
| `entities.description` | 实体固有属性 | ⚠️ 当前未使用，可暂留备用 |
| `event_entities.description`(role) | 精排时角色展示 | ❌ 删后精排丢失角色信息 |
| `event_entities.weight` | 实体权重 | ⚠️ 固定 1.0，当前未参与计算 |

### 7.2 关键认知澄清

| 易混淆点 | 正确事实 |
|---------|---------|
| `event_entities.description` 存的是什么？ | **存的是 `entity.role`（角色），不是 `entity.description`（固有属性）** |
| `entities.description` 和 `event_entities.description` 区别？ | 前者是固有属性（不随事件变化），后者是角色（随事件变化） |
| `entities.description` 在检索中使用吗？ | ❌ 当前流程未读取 |
| `event_entities.weight` 有作用吗？ | ⚠️ 固定 1.0，当前未参与计算 |

### 7.3 设计意图与现状对比

| 字段 | 设计意图 | 当前现状 | 建议 |
|------|---------|---------|------|
| `entities.description` | 实体消歧/精排补充 | 未使用 | 可在精排时补充实体固有属性 |
| `event_entities.weight` | 实体权重区分 | 固定 1.0 | 可基于共现频率动态计算 |
| `event_entities.description` | 存储角色 | ✅ 正常使用 | 建议重命名为 `role` 消除歧义 |

### 7.4 一句话总结

> **`entities.name` 是实体标识，`normalized_name` 是去重键，`event_entities.description` 实际存的是角色（role）用于精排展示**——四者协作实现"实体召回→多跳扩展→精排角色标注"的实体路检索闭环。`entities.description` 当前未被使用，属于备用字段。
