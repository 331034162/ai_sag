# 存储后端替换指南

当前默认使用 **MySQL + ChromaDB**，替换为其他数据库需进行二次开发。两部分的替换成本差异较大：向量库已有完整抽象层，替换成本低；关系库目前直接绑定 MySQL，替换需先建抽象层。

---

## 向量库替换（ChromaDB → Milvus / Qdrant / FAISS / PGVector）— 成本较低 ✅

向量库已有完善的 `BaseVectorStore` 抽象基类 + `create_vector_store()` 工厂模式，所有调用方依赖抽象接口，仅需新增实现类 + 注册。

### 1.1 需管理的 5 个逻辑集合

向量库统一管理 5 个独立 collection，新后端必须全部支持：

| Collection | 向量内容 | metadata | 用途 |
|-----------|---------|----------|------|
| `chunks` | 原始文档 chunk 文本 | `source_id` | Path B 语义兜底召回 |
| `event_titles` | 事件标题 | `source_id` | Path B 种子事件向量召回（title 路） |
| `event_contents` | 事件原文 chunk | `source_id` | BFS 新跳事件内容相似度计算 |
| `event_summaries` | 事件摘要（LLM 生成） | `source_id` | Path B 种子事件向量召回（summary 路，受 `embed_summary` 开关控制） |
| `entities` | 实体名称 | 无 | Path A 实体向量扩展（同义词近邻） |

> 所有 collection 的 id 规则：
> - `chunks` / `event_titles` / `event_contents` / `event_summaries`：`{source_id}:{chunk_idx}` 或 `evt:{event_id}`
> - `entities`：`ent:{entity_id}`

### 1.2 必须实现的 8 个同步抽象方法

[base.py](../vector_store/base.py) 定义了 8 个 `@abstractmethod`，新后端必须全部实现：

| 序号 | 方法签名 | 说明 |
|------|---------|------|
| 1 | `add(name, ids, texts, embeddings, metadatas=None)` | 批量写入向量（单 collection） |
| 2 | `query(name, query_embedding, top_k, similarity_threshold=0.0, source_ids=None)` | 相似度查询，返回 `[(id, score), ...]` |
| 3 | `delete_by_source(source_id)` | 按 source_id 删除所有相关向量（文档删除时调用） |
| 4 | `delete_entities_by_ids(entity_ids)` | 按 entity_id 批量删实体向量（对账清理孤儿实体） |
| 5 | `delete_event_ids(event_ids)` | 按 event_id 批量删事件向量（对账硬删软删事件） |
| 6 | `get_embeddings(name, ids)` | 按 id 批量取已存向量，返回 `{id: embedding}` |
| 7 | `list_source_ids()` | 列出所有已知 source_id（从 chunks collection 提取） |
| 8 | `list_all_entity_ids()` | 列出 entities collection 中所有 entity_id（供对账） |

**异步方法无需重写**：基类默认用 `asyncio.to_thread` 包装同步方法，子类如有原生异步能力（如 Milvus async）可选择性重写 `a*` 方法提升性能。

**语义化便捷方法无需重写**：`add_chunks` / `add_events` / `add_event_summaries` / `aquery_entities` 等方法已在基类实现，直接复用即可。

### 1.3 新增后端的步骤（共 4 步）

| 序号 | 文件 | 修改内容 | 工作量 |
|------|------|---------|--------|
| 1 | `vector_store/milvus_store.py`（示例） | **新建** — 继承 `BaseVectorStore` 实现上述 8 个抽象方法 | 中~高 |
| 2 | `vector_store/factory.py` | **修改** — 在 `create_vector_store()` 中注册新后端分支 | 极低（~5 行） |
| 3 | `vector_store/__init__.py` | **修改** — 导出新实现类 | 极低（1 行） |
| 4 | `base/config.py` | **修改** — `VectorStoreConfig` 增加对应连接参数（host/port/collection 等） | 低 |

> **重要**：`ingest/pipeline.py`、`retrieval/sag_retriever.py`、`retrieval/qa_engine.py` 等**所有调用方文件无需修改**，因为它们只依赖 `BaseVectorStore` 抽象接口。这是抽象层带来的核心收益。

## 关系库替换（MySQL → PostgreSQL / SQLite / SQL Server）— 成本较高 ⚠️

关系库目前**没有抽象接口层**，所有调用方直接依赖 `MysqlStore` 具体类。替换需先建立抽象层，再实现新后端。

### 2.1 替换步骤（共 10 步）

| 序号 | 文件 | 修改内容 | 工作量 |
|------|------|---------|--------|
| 1 | `storage/base.py` | **新建** — 定义 `BaseRelationalStore` 抽象基类，声明所有公开方法签名 | 中 |
| 2 | `storage/postgres_store.py`（示例） | **新建** — 用 `asyncpg` / `psycopg2` 实现 `BaseRelationalStore`，适配目标数据库 | **高** |
| 3 | `storage/schema_pg.sql`（示例） | **新建** — 目标数据库版 DDL 建表语句（含索引、约束、自动时间戳） | 低~中 |
| 4 | `storage/factory.py` | **新建** — `create_relational_store(cfg)` 工厂函数，按配置返回对应实现 | 低 |
| 5 | `storage/__init__.py` | **修改** — 导出工厂函数和新实现类 | 极低 |
| 6 | `storage/mysql_store.py` | **修改** — 让 `MysqlStore` 继承 `BaseRelationalStore`，保持行为不变 | 低 |
| 7 | `base/config.py` | **修改** — 改造 `MysqlConfig` 为通用 `RelationalConfig` 或新增 `PostgresConfig` | 中 |
| 8 | `ingest/pipeline.py` | **修改** — `from ..storage import MysqlStore` 改用工厂创建实例 | 低 |
| 9 | `retrieval/sag_retriever.py` | **修改** — 类型注解 `MysqlStore` → `BaseRelationalStore`，实例化改用工厂 | 低 |
| 10 | `retrieval/qa_engine.py` | **修改** — import + 类型注解 + 实例化改用工厂 | 低 |

### 2.3 MySQL 特有语法适配清单（共 9 处）

经核实 [mysql_store.py](../storage/mysql_store.py) 和 [schema.sql](../storage/schema.sql)，共有 9 处 MySQL 特有语法需要适配：

| 序号 | MySQL 语法 | 出现位置 | 行数 | PostgreSQL 等价写法 |
|------|-----------|---------|------|-------------------|
| 1 | `ON DUPLICATE KEY UPDATE` | `upsert_source` / `upsert_events`（事件表）/ `upsert_events`（实体表） | 3 处 | `INSERT ... ON CONFLICT (pk) DO UPDATE SET ...` |
| 2 | `INSERT IGNORE INTO` | `aisag_event_entities` 关联表（去重插入事件-实体关联） | 2 处 | `INSERT ... ON CONFLICT DO NOTHING` |
| 3 | `JSON_SET(metadata, '$.key', val)` | `mark_source_synced` 更新 metadata 字段 | 1 处 | `jsonb_set(metadata::jsonb, '{key}', to_jsonb(val), true)` |
| 4 | `JSON_EXTRACT(metadata, '$.vector_synced')` | 查询未同步的 source | 1 处 | `(metadata::jsonb ->> 'vector_synced')::int` |
| 5 | `ON UPDATE CURRENT_TIMESTAMP` | `schema.sql` 中 `updated_at` 字段自动更新时间戳 | 2 处 | 需用触发器（Trigger）实现，或在应用层手动维护 |

> **注意**：PostgreSQL 的 `JSONB` 性能优于 MySQL 的 `JSON`，迁移时可考虑将 `metadata` 字段类型从 `TEXT`（JSON 字符串）改为 `JSONB` 并加 GIN 索引，但需同步修改读写逻辑。

### 2.4 关键设计注意事项

1. **异步驱动**：当前 `MysqlStore` 使用 `aiomysql` + `asyncio` 全异步实现，新后端务必使用对应异步驱动（PostgreSQL 用 `asyncpg`，不要用同步 `psycopg2`），否则会阻塞事件循环。
2. **事务原子性**：`upsert_events` 中 事件插入 → 实体插入 → 关联表插入 是在一个事务中完成的，新后端需保证相同的原子性语义。
3. **批量操作**：`get_events_for_entities`、`get_entity_degrees` 等方法使用了 `IN (...)` 批量查询，新后端需注意参数数量上限（PostgreSQL 默认 32767，足够；SQLite 默认 999，需分批）。
4. **软删除语义**：当前 MySQL 实现用 `deleted_at IS NULL` 表示未删除，新后端需保持相同语义，`soft_delete_source` 只标记 `deleted_at` 不真正删数据，`hard_delete_*` 才真正删除。

## 快速验证清单（替换后）

无论替换向量库还是关系库，完成后请按以下清单验证：

- [ ] API 服务正常启动，无 import 错误
- [ ] `GET /api/health` 返回 200
- [ ] `GET /api/stats` 返回正确统计
- [ ] 能成功上传一个 md/txt 文档
- [ ] 上传后文档出现在 `GET /api/documents` 列表中
- [ ] 上传后能立即搜索到文档内容（全文搜索 + 向量搜索）
- [ ] 多跳问答能正常返回结果，trace 日志中无异常
- [ ] 删除文档后，`GET /api/documents` 中不再出现
- [ ] 删除文档后重启服务，数据一致性正常（对账不报错）
