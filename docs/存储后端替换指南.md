# 存储后端替换指南

当前支持 **PostgreSQL / MySQL**（关系库）+ **PGVector / ChromaDB / FAISS / Milvus**（向量库），均可通过 `.env` 配置切换，无需改代码。

> **推荐部署**：PostgreSQL + pgvector，一套数据库同时承担关系库 + 向量库，无需安装多个中间件，部署运维最简单。
>
> **其他组合**：
> - 关系库用 MySQL + 向量库用 ChromaDB/FAISS/Milvus（需分别部署两个中间件）
> - 关系库用 PostgreSQL + 向量库用 ChromaDB/FAISS/Milvus（PG 只做关系库）

---

## 一、关系库切换（PostgreSQL ↔ MySQL）— 已支持 ✅

关系库已有 `PgStore`（推荐）/ `MysqlStore` 两个实现，通过 `create_db_store(cfg)` 工厂按 `AISAG_DB_BACKEND` 配置自动选择。

### 1.1 配置切换

```ini
# .env 中切换
AISAG_DB_BACKEND=postgresql   # postgresql（推荐）| mysql

# PostgreSQL 连接参数（推荐）
SAG_PG_HOST=localhost
SAG_PG_PORT=5432
SAG_PG_DATABASE=sag
SAG_PG_USER=postgres
SAG_PG_PASSWORD=your_password
```

前置条件（PostgreSQL）：
```sql
CREATE DATABASE sag;
-- 如使用 pgvector 后端，还需安装扩展：
CREATE EXTENSION IF NOT EXISTS vector;
```
表结构首次启动自动创建（[schema_pg.sql](../storage/schema_pg.sql)），无需手动建表。

### 1.2 两个实现的位置

| 后端 | 实现文件 | DDL | 驱动 |
|------|---------|-----|------|
| PostgreSQL（推荐） | [storage/pg_store.py](../storage/pg_store.py) | [storage/schema_pg.sql](../storage/schema_pg.sql) | asyncpg（异步） |
| MySQL | [storage/mysql_store.py](../storage/mysql_store.py) | [storage/schema.sql](../storage/schema.sql) | aiomysql（异步） |

### 1.3 公共接口

两个实现都继承自 `BaseRelationalStore`（[storage/base.py](../storage/base.py)），对外接口完全一致。`create_db_store()` 工厂（[storage/factory.py](../storage/factory.py)）按配置返回对应实现。

所有调用方（`ingest/pipeline.py`、`retrieval/sag_retriever.py`、`retrieval/qa_engine.py`）只依赖 `BaseRelationalStore` 抽象接口，**切换后端无需修改业务代码**。

### 1.4 FAISS 后端的映射表

FAISS 向量库需要通过 `faiss_*_map` 映射表反查 `faiss_hash → UUID`。两个 Store 都已实现以下公共方法，供 FAISS 后端调用：

- `fetch_chunks_by_hashes(hashes)` / `fetch_events_by_hashes(hashes)` / `fetch_entities_by_hashes(hashes)`
- `fetch_source_ids_by_hashes(hashes)`
- `fetch_hashes_by_source(map_table, source_id)`
- `fetch_hashes_by_source_document(map_table, source_id, document_id)`
- `fetch_all_hashes(map_table)`
- `fetch_distinct_source_ids()` / `fetch_all_entity_ids()`

系统在 FAISS 后端时自动启用映射表写入（`faiss_map_enabled=True`），其他向量库后端不写映射表。

## 二、向量库替换 — 已支持 ✅

向量库已有完善的 `BaseVectorStore` 抽象基类 + `create_vector_store()` 工厂模式，所有调用方依赖抽象接口，仅需新增实现类 + 注册。

### 2.1 已实现的后端

| 后端 | 实现文件 | 说明 |
|------|---------|------|
| PGVector | [vector_store/pgvector_store.py](../vector_store/pgvector_store.py) | **推荐**，PostgreSQL + pgvector 扩展，与关系库合一部署 |
| ChromaDB | [vector_store/chroma_store.py](../vector_store/chroma_store.py) | 本地轻量持久化 |
| FAISS | [vector_store/faiss_store.py](../vector_store/faiss_store.py) | 单机内存索引 + 本地持久化，需配合关系库映射表 |
| Milvus | [vector_store/milvus_store.py](../vector_store/milvus_store.py) | 分布式向量库 |

### 2.2 配置切换

```ini
# .env 中切换
AISAG_VECTOR_STORE_BACKEND=pgvector   # pgvector（推荐）| chroma | faiss | milvus

# PGVector 特有配置（推荐：与关系库同一个 PG 实例）
AISAG_PG_CONNECTION_STRING=postgresql://postgres:your_password@localhost:5432/sag
# HNSW 索引参数（仅 AISAG_PG_HNSW_INDEX=true 时生效）
# AISAG_PG_HNSW_M=16                   # 每节点最大连接数（默认 16）
# AISAG_PG_HNSW_EF_CONSTRUCTION=64     # 建索引时候选列表大小（默认 64）
# AISAG_PG_HNSW_EF_SEARCH=40           # 查询时候选列表大小（默认 40）

# 其他后端特有配置见 .env.example
```

> **PGVector 表名说明**：LlamaIndex 库会自动在表名前加 `data_` 前缀，实际表名为 `data_{prefix}{collection}`。例如 prefix=`sag_` 时，5 张表为 `data_sag_chunks` / `data_sag_event_titles` / `data_sag_event_contents` / `data_sag_event_summaries` / `data_sag_entities`。可通过 psql 直接查询验证。

### 2.3 需管理的 5 个逻辑集合

向量库统一管理 5 个独立 collection，所有后端必须全部支持：

| Collection | 向量内容 | metadata | 用途 |
|-----------|---------|----------|------|
| `chunks` | 原始文档 chunk 文本 | `source_id` | Path B 语义兜底召回 |
| `event_titles` | 事件标题 | `source_id` | Path B 种子事件向量召回（title 路） |
| `event_contents` | 事件原文 chunk | `source_id` | BFS 新跳事件内容相似度计算 |
| `event_summaries` | 事件摘要（LLM 生成） | `source_id` | Path B 种子事件向量召回（summary 路，受 `embed_summary` 开关控制） |
| `entities` | 实体名称 | 无 | Path A 实体向量扩展（同义词近邻） |

### 2.4 FAISS 后端的特殊性

FAISS 的 `IndexIDMap2` 要求 id 为 int64，无法直接存 UUID。处理方式：

1. 通过 `blake2b` 把 UUID 哈希为 int64（`storage.uuid_to_int64`），存入 FAISS
2. 关系库维护 `faiss_*_map` 映射表（`faiss_hash ↔ uuid/source_id/document_id`）
3. 查询时：FAISS 返回 hash → 关系库 `fetch_*_by_hashes` 反查 UUID
4. 删除时：从映射表反查 hash → `faiss.remove_ids` 硬删除

FAISS 后端通过 `db_store` 参数接收关系库实例（`create_vector_store(cfg, db_store=db)`），不直接拼 SQL，与 MySQL/PG 后端无关。

### 2.5 新增向量库后端的步骤

| 序号 | 文件 | 修改内容 | 工作量 |
|------|------|---------|--------|
| 1 | `vector_store/xxx_store.py` | **新建** — 继承 `BaseVectorStore` 实现 8 个抽象方法 | 中~高 |
| 2 | `vector_store/factory.py` | **修改** — 在 `create_vector_store()` 中注册新后端分支 | 极低（~5 行） |
| 3 | `vector_store/__init__.py` | **修改** — 导出新实现类 | 极低（1 行） |
| 4 | `base/config.py` | **修改** — `VectorStoreConfig` 增加对应连接参数 | 低 |

> **重要**：`ingest/pipeline.py`、`retrieval/sag_retriever.py`、`retrieval/qa_engine.py` 等**所有调用方文件无需修改**。

## 三、快速验证清单（切换后端后）

无论切换关系库还是向量库，完成后请按以下清单验证：

- [ ] API 服务正常启动，无 import 错误
- [ ] `GET /api/health` 返回 200
- [ ] `GET /api/stats` 返回正确统计
- [ ] 能成功上传一个 md/txt 文档
- [ ] 上传后文档出现在 `GET /api/documents` 列表中
- [ ] 上传后能立即搜索到文档内容（全文搜索 + 向量搜索）
- [ ] 多跳问答能正常返回结果，trace 日志中无异常
- [ ] 删除文档后，`GET /api/documents` 中不再出现
- [ ] 删除文档后重启服务，数据一致性正常（对账不报错）
- [ ] FAISS 后端额外验证：`verify_integrity()` 对账通过（FAISS hash 与映射表一致）
