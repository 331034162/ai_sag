## 二、历史在检索阶段的传导机制

历史在检索阶段**只直接作用于查询重写**，产出的 `eff_query` 像一根线，把历史信息传导到所有下游环节。

### 2.1 传导链路图

数据源图例：
- `[计算]`   纯本地计算（Embedder 编码），不访问存储
- `[LLM]`    大模型调用
- `[向量库]` Chroma 向量数据库（ANN 检索）
- `[数据库]` MySQL 关系数据库（结构化查询）

```
history ──┐
          │
query ────┤
          ▼
   ┌─────────────────────┐
   │ 0. 查询重写 [LLM]    │  ← 历史 + query → eff_query
   └──────────┬──────────┘
              │ eff_query
              │
   ┌──────────┴────────────────────────────────────────────────────────┐
   │ 1. 向量编码  [计算]    eff_query → query_vec                       │
   │ 2. 实体抽取  [LLM]     eff_query → query_entities                 │
   │ 3. 标题召回  [向量库]   query_vec → title_event_ids                │
   │ 4. 实体召回  [数据库]   query_entities → entity_event_ids          │
   │ 5. 多跳扩展  [数据库]   seed_ids → expanded_ids（与历史无关）       │
   │ 6. 粗排     [向量库]   query_vec + expanded_ids → ranked_ids     │
   │ 7. LLM 精排 [LLM]      eff_query + ranked_ids → reranked_ids      │
   │ 8. 切片回取 [数据库]   reranked_ids → sections（与历史无关）       │
   │ 9. 双路融合 [向量库]   query_vec + sections → 最终 sections       │
   └────────────────────────────────────────────────────────────────────┘
```

**数据流向说明**：
- eff_query 先经 `[计算]` 生成 query_vec，再经 `[LLM]` 抽取 query_entities；
- 召回分两路并行：标题走 `[向量库]`、实体走 `[数据库]`；
- `[数据库]` 负责"图结构"类操作（实体反查事件、多跳扩展、回取原文）；
- `[向量库]` 负责"语义相似"类操作（标题召回、内容粗排、切片补召）。

### 2.2 各环节历史参与度对比

**"历史参与方式"三类定义**：
- **直接**：操作直接读取 `history` 参数（环节 0、10）
- **间接**：操作的输入是 `eff_query` 的衍生物（query_vec / query_entities / 候选事件）
- **无**：操作不读 history，输入也不是 eff_query 衍生物；但输入是上游环节的结果，**数据流层面仍间接受历史影响**

| 环节 | 数据源 | 历史参与方式 | 作用强度 | 失效后果 |
|------|--------|------------|---------|---------|
| 0. 查询重写 | `[LLM]` | **直接** | ★★★★★ | eff_query 错误，全链路失效 |
| 1. 实体抽取 | `[LLM]` | 间接（eff_query） | ★★★★ | 抽不到正确实体，图谱路径失效 |
| 2. 向量编码 | `[计算]` | 间接（eff_query） | ★★★★ | 向量偏移，召回不准 |
| 3. 标题召回 | `[向量库]` | 间接（query_vec） | ★★★ | 召回不到相关事件 |
| 4. 实体召回 | `[数据库]` | 间接（query_entities） | ★★★ | 图谱种子缺失 |
| 5. 多跳扩展 | `[数据库]` | 无（输入 seed_ids 受上游影响） | ★ | 扩展范围由种子决定 |
| 6. 粗排 | `[向量库]` | 间接（query_vec） | ★★ | 排序偏移，相关事件被截断 |
| 7. LLM 精排 | `[LLM]` | 间接（eff_query） | ★★★★ | LLM 不理解真实意图，排序混乱 |
| 8. 切片回取 | `[数据库]` | 无（输入 reranked_ids 受上游影响） | - | 与查询无关 |
| 9. 双路融合 | `[向量库]` | 间接（query_vec） | ★★ | 补充切片不相关 |
| 10. 答案生成 | `[LLM]` | **直接** | ★★★★★ | 答案不连贯，代词无法消解 |

**回答你的疑问**：
- 标 **直接** 或 **间接** 的环节（0,1,2,3,4,6,7,9,10）：历史都参与了 ✅
- 标 **无** 的环节（5 多跳扩展、8 切片回取）：操作本身不使用 history/eff_query，但其输入（seed_ids / reranked_ids）来自上游，**数据流层面仍间接受历史影响**，只是影响是"被动传导"的，不是这个环节主动读取历史

### 2.3 数据源分布统计

| 数据源 | 环节数 | 环节列表 | 主要职责 |
|--------|-------|---------|---------|
| `[LLM]` | 4 | 0,1,7,10 | 查询重写、实体抽取、精排、答案生成 |
| `[计算]` | 1 | 2 | 向量编码 |
| `[向量库]` | 3 | 3,6,9 | 标题召回、内容粗排、切片补召 |
| `[数据库]` | 3 | 4,5,8 | 实体反查事件、多跳扩展、回取原文 |

**关键洞察**：
- 9 个检索环节（1-9）中，**向量库 3 个、数据库 3 个、LLM 2 个、计算 1 个**，两库并重，没有单一存储承担全部召回；
- **召回类（环节 3/4/9）**：向量库负责语义召回（标题、切片），数据库负责结构召回（实体→事件）；
- **数据库承担的 3 个环节（4/5/8）中，环节 4（实体召回）历史间接参与（输入 query_entities），环节 5、8 历史不直接参与（纯结构操作）**；
- **历史真正"无作用"的只有 2 个环节（5 多跳扩展、8 切片回取）**，都是数据库的纯结构操作（BFS 遍历、按 ID 取原文），与查询语义无关。

## 三、历史在两个关键环节的直接使用

### 3.1 检索阶段：查询重写

**目的**：把依赖上下文的"指代型追问"重写为独立可检索的查询。

核心代码（sag_retriever.py）：
```python
async def _rewrite_query(self, query: str, history: list[dict]) -> str:
    if not history:
        return query
    max_rounds = self.cfg.search.rewrite_max_rounds  # 可配置，默认 5
    history_text = self._format_history(history, max_rounds)
    if not history_text:
        return query
    return await self._llm_rewrite(history_text, query)
```

**历史被"压扁"为纯文本**：
```python
@staticmethod
def _format_history(history, max_rounds):
    valid = [h for h in history
             if h.get("role") in ("user", "assistant") and h.get("content", "").strip()]
    max_msgs = max_rounds * 2  # 5轮=10条
    if len(valid) > max_msgs:
        valid = valid[-max_msgs:]  # 保留最近5轮
    role_label = {"user": "用户", "assistant": "助手"}
    return "\n".join(f"{role_label[h['role']]}：{h['content']}" for h in valid)
```

**重写效果示例**：

| 用户原话 | 历史上下文 | 重写后（eff_query） |
|---------|-----------|-------------------|
| "那它的担保方式呢？" | 上一轮聊过"众商贷" | "众商贷的担保方式是什么？" |
| "和众微贷比呢？" | 上一轮聊过"众商贷担保方式" | "众商贷与众微贷的担保方式对比" |
| "利率多少？" | 上一轮聊过"众商贷" | "众商贷的利率是多少？" |
| "再详细说说" | 上一轮聊过"众商贷担保方式" | "众商贷担保方式的详细信息" |

### 3.2 生成阶段：消息组装

**目的**：让 LLM 生成答案时能理解上下文（代词、省略主语、已提及信息）。

核心代码（qa_engine.py）：
```python
def _build_messages(self, query, result, history):
    context = self._build_context(result) if result.sections else "（未检索到相关资料）"
    system_content = CHAT_SYSTEM_PROMPT.format(context=context)
    messages = [ChatMessage(role=SYSTEM, content=system_content)]

    truncated = self._truncate_history(history, MAX_HISTORY_ROUNDS=5)
    role_map = {"user": MessageRole.USER, "assistant": MessageRole.ASSISTANT}
    for h in truncated:
        role = role_map.get(h.get("role", "user"), MessageRole.USER)
        content = h.get("content", "")
        if content:
            messages.append(ChatMessage(role=role, content=content))

    messages.append(ChatMessage(role=MessageRole.USER, content=query))
    return messages
```

**最终送入 LLM 的消息结构**：
```
[0] SYSTEM    ← CHAT_SYSTEM_PROMPT + 本次检索的【片段1..N】资料
[1] USER      ← 历史 round 1 user
[2] ASSISTANT ← 历史 round 1 assistant
[3] USER      ← 历史 round 2 user
[4] ASSISTANT ← 历史 round 2 assistant
...
[N-1] USER    ← 当前 query（用原始 query，不是 eff_query）
```

## 四、两个阶段的使用对比

| 维度 | 检索阶段（查询重写） | 生成阶段（消息组装） |
|------|---------------------|---------------------|
| **历史格式** | 纯文本拼接（"用户：xxx\n助手：xxx"） | 原生多轮 ChatMessage 列表 |
| **历史角色** | 渲染为标签文本 | 映射为 USER/ASSISTANT |
| **资料携带** | ❌ 不带 | ✅ 当前轮资料放 SYSTEM |
| **截断轮数** | 5 轮（10 条消息） | 5 轮（10 条消息） |
| **调用方式** | 单次 `achat` | 单次 `achat` / 流式 `astream_chat` |
| **失败兜底** | 回退原 query | 异常向上抛 |
| **使用的 query** | eff_query（重写后） | 原始 query（未重写） |
| **目的** | 消解代词 → 独立可检索 query | 让 LLM 理解上下文生成连贯答案 |

### 为什么检索用 eff_query，生成用原始 query？

- **检索阶段用 `eff_query`**：检索系统需要"独立完整的查询"才能精准匹配，"那它的担保方式呢？"无法直接检索；
- **生成阶段用 `query` + `history`**：LLM 有能力从原始表达 + 历史上下文中理解意图，且保留原始表达能让答案更自然（用户说"它"，LLM 也用"它"回答，而不是生硬地说"众商贷的担保方式是..."）。

## 六、资料不进历史的设计

### 6.1 设计原则

| 内容 | 是否进入历史 | 原因 |
|------|------------|------|
| 用户问题文本 | ✅ | 理解上下文必需 |
| 助手答案文本 | ✅ | 延续上下文必需 |
| 检索的资料（片段） | ❌ | 每轮重新检索，避免 token 爆炸 |
| 检索的 trace | ❌ | 仅当前轮调试用 |

### 6.2 Token 预算对比

假设每轮检索 5 个片段，每片段 512 字，单轮答案 200 字：

| 策略 | 单轮 token | 5 轮历史 token | 总计 |
|------|-----------|--------------|------|
| 资料进历史 | 5×512 + 200 = 2760 | 5 × 2760 = 13800 | 13.8K（逼近上下文上限） |
| 资料不进历史 | 200 | 5 × 200 = 1000 | 1K（+ 当前轮资料 2560 = 3.56K） |

## 七、前端历史维护逻辑

前端维护 `chatHistory` 数组，**只在本轮成功后才入历史**（避免失败时污染上下文）：

```javascript
// 普通对话
const data = await api('/api/chat', {
  method:'POST', headers:{'Content-Type':'application/json'},
  body: JSON.stringify({ query, history: chatHistory, ... })
});
// 成功后才把本轮 user + assistant 一起入历史
chatHistory.push({ role: 'user', content: query });
chatHistory.push({ role: 'assistant', content: answer });
```

## 八、完整实例：3 轮对话的全链路分析

### 8.1 对话历史

| 轮次 | 角色 | 内容 |
|------|------|------|
| Round 1 | 用户 | "众商贷利率多少？" |
| Round 1 | 助手 | "众商贷年化利率为 7.2%..." |
| Round 2 | 用户 | "那它的担保方式呢？" |
| Round 2 | 助手 | "众商贷是信用贷款，无需担保..." |
| Round 3 | 用户 | "和众微贷比呢？" ← **当前轮** |

### 8.2 Round 3 的全链路处理

| 环节 | 输入 | 输出 | 历史的作用 |
|------|------|------|----------|
| 0. 重写 | "和众微贷比呢？" + history | eff_query="众商贷与众微贷担保方式对比" | 消解"和...比"省略的对比对象 |
| 1. 实体抽取 | eff_query | [众商贷, 众微贷, 担保方式] | 抽到两个产品实体 |
| 2. 向量编码 | eff_query | query_vec | 向量指向"两产品担保对比"语义 |
| 3. 标题召回 | query_vec | 相关事件 | 召回"众商贷担保""众微贷担保"事件 |
| 4. 实体召回 | [众商贷, 众微贷] | entity_event_ids | 反查两产品关联的所有事件 |
| 5. 多跳扩展 | seed_ids | 扩展事件 | 沿实体边扩展（无历史直接作用） |
| 6. 粗排 | query_vec | ranked_ids | 按"两产品担保对比"语义排序 |
| 7. 精排 | eff_query | reranked_ids | LLM 理解要"对比"，优先选对比类事件 |
| 8. 切片回取 | event_ids | sections | 取出原文（无历史作用） |
| 9. 双路融合 | query_vec | 最终 sections | 补充向量相似的切片 |
| 10. 生成 | "和众微贷比呢？" + history + sections | "众商贷与众微贷担保方式对比如下..." | LLM 理解"和...比"是对比意图 |

### 8.3 如果没有历史，Round 3 会怎样？

| 场景 | 无历史 | 有历史 |
|------|--------|--------|
| 查询重写 | eff_query="和众微贷比呢？" | eff_query="众商贷与众微贷担保方式对比" |
| 实体抽取 | [众微贷]（漏了对比对象） | [众商贷, 众微贷, 担保方式] |
| 标题召回 | 召回一堆"众微贷"事件（不知道比什么） | 精准召回两产品担保对比事件 |
| 精排 | LLM 不知道要对比，排序混乱 | LLM 明确知道要对比，优先选对比类事件 |
| 答案生成 | "众微贷的担保方式是..."（答非所问） | "众商贷与众微贷担保方式对比如下..."（精准回答） |

## 九、核心配置参数汇总

| 参数 | 默认值 | 环境变量 | 作用 |
|------|--------|----------|------|
| `rewrite_max_rounds` | 5 | `AISAG_REWRITE_MAX_ROUNDS` | 查询重写保留的历史轮数 |
| `MAX_HISTORY_ROUNDS` | 5 | 硬编码 | 生成阶段保留的历史轮数 |
| `AISAG_LLM_TIMEOUT` | 120s | 环境变量 | LLM 调用超时（含重写） |
| `AISAG_LLM_MAX_RETRIES` | 3 | 环境变量 | LLM 重试次数（重写失败回退原 query） |

### 配置建议

| 场景 | 建议轮数 | 原因 |
|------|---------|------|
| 短问答（FAQ） | 3 轮 | 上下文需求小，节省 token |
| 长对话（咨询） | 5 轮（默认） | 平衡上下文与 token |
| 复杂推理 | 8-10 轮 | 需要更多上下文，但注意 token 上限 |

## 十二、总结

多轮对话历史的处理可以概括为 **"两次直接参与 + 全链路传导"**：

1. **检索阶段**：历史**只直接作用于查询重写**，产出的 `eff_query` 像一根线，把历史信息**传导**到实体抽取、向量编码、召回、排序等所有下游环节，是"一夫当关"的守门员；
2. **生成阶段**：历史**直接作为多轮消息**送入 LLM，让 LLM 理解代词、省略、已提及信息，输出连贯答案；
3. **不参与的环节**：多跳扩展、切片回取是纯结构操作，与查询语义无关，历史不影响它们；
4. **资料不进历史**：每轮重新检索当前 query 的资料，避免 token 爆炸，保证资料始终与当前问题对齐。

这种"重写传导 + 生成直用"的双层设计，既保证了检索的精准性，又保证了生成的自然性。
