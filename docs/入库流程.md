## 一、整体架构

ai_sag 入库严格遵循 SAG 论文离线建库阶段：**不建静态图谱，只建事件-实体多对多索引**。

```
原始文档 → 加载 → 清洗 → 切分 → 文体识别 → LLM抽取(事件+实体) → 混合存储(MySQL + 向量库)
```

### 核心原则

- **Chunk ↔ Event 严格 1:1**：一个 chunk 抽一个融合事件，不拆三元组
- **实体只做索引锚点**：不存储完整语义，通过 `event_entities` 关联表串联跨文档事件
- **实体全局去重**：按 `(type, normalized_name)` 跨文档共享
- **入库=查询口径一致**：11 类实体类型、命名规范在入库和查询抽取中共用同一份常量
- **全链路异步**：MySQL 用 aiomysql 连接池，LLM 用 LlamaIndex 原生异步，Embedder/VectorStore 用 `asyncio.to_thread` 包装

### 流水线全景


## 二、入库流程详解

### 阶段 0：服务初始化

`IngestPipeline.init()` 在 FastAPI 启动时调用：

```
IngestPipeline.__init__(cfg)
  ├─ loader = DocumentLoader.default()      # 文档加载器
  ├─ cleaner = TextCleaner()                 # 文本清洗器
  ├─ embedder = create_embedder(cfg)         # Embedding 模型
  ├─ splitter = create_splitter(cfg, embed_model)  # 复用 embedder._model（semantic 模式需要）
  ├─ llm_factory = LlmFactory(cfg)          # LLM 工厂（按场景取 LLM）
  ├─ llm = llm_factory.get("GENRE_CLASSIFY") # 体裁分类用 LLM
  ├─ extractor = EventExtractor(llm, ...)    # 事件抽取器
  ├─ db = MysqlStore(...)                    # MySQL 异步连接池
  ├─ vectors = create_vector_store(cfg)      # 向量库（ChromaDB）
  ├─ _reconcile_task = None                  # 对账任务占位
  └─ _ingest_semaphore = Semaphore(concurrency=2)  # 并发入库信号量

await pipeline.init()
  ├─ db.ensure_schema()                      # 建表（如不存在）
  ├─ _reconcile()                            # 启动时对账一次（四类清理）
  └─ _start_reconcile_loop()                 # 启动定时对账后台任务
```

### 阶段 1：文档加载（Loader）

`DocumentLoader.default()` 注册 5 个 Reader，按文件后缀分发：

| Reader | 支持后缀 | 依赖库 | 特殊处理 |
|--------|----------|--------|----------|
| `MarkdownReader` | `.md` `.markdown` | 内置 `open` | UTF-8 读取，`errors="ignore"` 容错 |
| `TextReader` | `.txt` `.text` `.log` | 内置 `open` | 同上 |
| `DocxReader` | `.docx` | `python-docx` | 遍历所有段落，按 `\n\n` 拼接 |
| `PDFReader` | `.pdf` | `PyMuPDF (fitz)` | 逐页提取文本，按页拼接 |
| `ExcelReader` | `.xlsx` | `openpyxl` + `doc_parser/excel/v2` | 解析为 Markdown 表格，含防御性校验 |

**Excel 入口防御性校验**（[readers.py](file:///d:/ai_sag_git/ai_sag/loader/readers.py)）：
1. 存在性校验：文件不存在直接抛错；
2. 扩展名拦截：`.xls`（旧二进制格式）要求先转 `.xlsx`；
3. 大小上限：100MB，超过拒绝；
4. 签名校验：读取文件头 4 字节，必须是 `PK\x03\x04`（xlsx 本质是 zip），防止后缀欺骗；
5. 样式表容错：`_load_workbook_safe` 在 openpyxl 解析 styles 失败时替换为最小有效 `styles.xml`（预填充 1000 个空 `<xf/>`），解决非标准 Excel 报错。

**文件上传流程**：
- FastAPI 通过 `UploadFile.file.read()` 读取字节；
- 写入临时文件 `tempfile.NamedTemporaryFile(delete=False, suffix=...)`；
- 调 `loader.load(tmp_path)` → `LoadedDocument`；
- `finally` 块中 `os.unlink(tmp_path)` 清理临时文件。

**纯文本入口**：`loader.load_text(title, content)` 直接构造 `LoadedDocument(file_type="text")`，跳过文件解析。

产物：`LoadedDocument(title, content, source_path, file_type, metadata)`

代码位置：[loader/base.py](file:///d:/ai_sag_git/ai_sag/loader/base.py)、[loader/readers.py](file:///d:/ai_sag_git/ai_sag/loader/readers.py)

### 阶段 2：文本清洗（Cleaner）

`TextCleaner.clean()` 按顺序执行 5 步正则清洗（可通过构造参数开关）：

| 步骤 | 默认开 | 正则/操作 | 作用 |
|------|--------|-----------|------|
| 去 HTML 标签 | ✅ | `<[^>]+> → ""` | 清除从 Word/PDF/网页残留的标签 |
| 统一换行 | ✅ | `\r\n` / `\r → \n` | 跨平台换行统一 |
| 规整空白 | ✅ | 全角空格 `\u3000`、连续空格/制表符 → 单空格；逐行 strip | 避免异常空白 |
| 合并硬断行 | ✅ | `(?<!\n)\n(?!\n) → " "` | PDF 复制常见的"每行被硬换行"问题 |
| 压缩空行 | ✅ | `\n{3,} → \n\n`，整体 strip | 连续空行压缩为一个空行 |

清洗后返回**新的** `LoadedDocument`，`metadata["cleaned"]=True`，不修改原对象。

代码位置：[cleaner/text_cleaner.py](file:///d:/ai_sag_git/ai_sag/cleaner/text_cleaner.py)

### 阶段 3：文档切分（Splitter）

切分器由 `create_splitter(cfg, embed_model)` 工厂根据 `cfg.splitter.mode` 创建：

| 模式 | 默认 | 说明 | 适用 |
|------|------|------|------|
| `semantic` | ✅ 默认 | `SemanticSplitterNodeParser`（percentile=95），按语义相似度断点切分 | chunk 语义最完整，**最适合 SAG 事件抽取** |
| `auto` | - | md→markdown，其余→sentence | 速度快、适配性强 |
| `markdown` | - | `MarkdownNodeParser` 按 `#` 标题切，超长节点用 SentenceSplitter 二级切 | 原生 Markdown 文档 |
| `sentence` | - | `SentenceSplitter(chunk_size, chunk_overlap)` | docx/pdf/txt 兜底 |
| `token` | - | `TokenTextSplitter` 按 token 精确切 | 严格 token 预算场景 |
| `code` | - | `CodeSplitter` 按语法结构（需配合 language） | 代码类文档 |

**默认配置**（可通过环境变量覆盖）：
- `AISAG_SPLITTER_MODE=semantic`
- `AISAG_CHUNK_SIZE=8192`
- `AISAG_CHUNK_OVERLAP=800`
- `AISAG_SPLITTER_LANGUAGE=python`
- `AISAG_BREAKPOINT_PERCENTILE=95`（semantic 模式断点分位阈值）

**关键细节**：
- semantic 模式需要 `embed_model`，直接复用 `IngestPipeline` 已构建的 `embedder._model`（避免重复加载模型）；
- 切分前包装为 LlamaIndex `Document(text=doc.content, metadata={"title": doc.title})`；
- 每个 Node 转换为 `Chunk`：
  - `id = uuid.uuid4()`；
  - `heading` 从 node.metadata 按 `Header_1 > Header_2 > Header_3 > header > title` 优先级取，都没有则 `"Introduction"`；
  - `rank_index` 按切分顺序递增（0-based）；
  - 空文本 chunk 直接丢弃。
- 切分后若 `chunks=[]`，直接抛 `ValueError("切分后无有效切片")` 终止入库。

产物：`list[Chunk]`

代码位置：[splitter/factory.py](file:///d:/ai_sag_git/ai_sag/splitter/factory.py)、[splitter/chunk_splitter.py](file:///d:/ai_sag_git/ai_sag/splitter/chunk_splitter.py)、[splitter/auto_splitter.py](file:///d:/ai_sag_git/ai_sag/splitter/auto_splitter.py)

### 阶段 4：文体识别（Genre Detect）—— 文体感知增强

入库前置步骤，用 LLM 判断文档文体，驱动后续抽取的判别规则和 role 词汇表。

**设计要点**：
- **文档级而非 chunk 级**：每文档仅 1 次 LLM 调用，降低判断误差和成本；
- **样本裁剪**：只取标题 + 首尾 800/500 字，避免全文输入浪费 token；
- **短路优化**：`xlsx/csv` 直接返回 `tabular`，省一次 LLM 调用；
- **输出枚举**：限定为 `SUPPORTED_GENRES`，防止 LLM 自创标签；
- **失败降级**：异常或开关关闭时返回 `generic`，不阻断入库；
- **查询侧不调用**：查询无文体属性，保证入库/查询口径一致。

**支持的文体**（`SUPPORTED_GENRES`）：

| 文体 | 含义 |
|------|------|
| `news` | 新闻报道 |
| `contract` | 合同/协议 |
| `financial_report` | 财务报告 |
| `legal` | 法律文书 |
| `meeting` | 会议纪要 |
| `manual` | 说明书/操作手册 |
| `academic` | 学术论文/研究报告 |
| `report` | 通用报告（工作/调研/年度/政务，非学术） |
| `fiction` | 小说/虚构叙事 |
| `poetry` | 诗歌/散文诗 |
| `tabular` | 表格数据 |
| `generic` | 通用兜底 |

开关：`AISAG_GENRE_DETECT=true`（默认开启）

代码位置：[pipeline.py:120-167](file:///d:/ai_sag_git/ai_sag/ingest/pipeline.py#L120-L167) `_detect_genre`、[prompts.py](file:///d:/ai_sag_git/ai_sag/base/prompts.py) `SUPPORTED_GENRES`

### 阶段 5：事件/实体抽取（Extractor）—— SAG 核心

`EventExtractor.extract_batch(chunks, doc_title, genre)` 把每个 chunk 送到 LLM，**抽取一个融合事件及其关联实体**。

#### 抽取模型（Pydantic）

```python
class _Entity(BaseModel):
    type: str          # 11 类实体之一
    name: str          # 归一化后的实体名
    description: str   # 实体固有属性（如"互联网银行"），不随事件变化
    role: str          # 在当前事件中的角色标签（4-10字，如"贷款方"/"签约时间"/"标的物")

class _Event(BaseModel):
    title: str
    summary: str       # ≤500 字摘要
    content: str       # 有机叙事主线（非要点列表）
    entities: list[_Entity]
```

#### Prompt 核心约束（`extract_system_prompt()`）

1. **强制单事件融合**：一个 chunk 只抽一个综合顶层事件，不拆成多个三元组，避免语义碎片化；
2. **全局扫描**：先识别时间/地点/主体/动作/对象/数据/因果/对比/关系，再写事件；
3. **11 类实体**（与查询侧完全一致）：`time/location/person/organization/group/topic/work/product/action/metric/label`；
4. **实体命名规范**：去"有限公司/股份公司/集团"等后缀、去地名修饰、时间用阿拉伯数字；
5. **description vs role 严格区分**：
   - `description`：实体**固有属性**，不随事件变化（例："互联网银行"）；
   - `role`：实体**在当前事件中的角色标签**，4-10 字短词（例："贷款方"、"被担保方"、"签约主体"、"标的物"）；
6. **跨片段关联**：顺序抽取时，前一片段的 `summary` 作为 `previous_context` 传入下一片段，消解代词和省略主语；
7. **文体感知**：`genre` 透传给 extractor，驱动方案 A+B 文体感知增强（不同文体用不同的 role 词汇表和判别规则）。

#### 两种抽取模式

| 模式 | 实现 | previous_context | 速度 | 跨片段消解 |
|------|------|------------------|------|------------|
| 顺序（默认 `false`） | `extract_batch` 单线程 for 循环 | ✅ 传递前一事件 summary | 慢 | 好 |
| 并行（`true`） | `asyncio.gather` + `run_in_executor`（workers=4） | ❌ 不传，仅用 doc_title 全局上下文 | 快 ~4x | 退化 |

```
顺序模式（默认，质量优先）：
chunk1 → LLM → event1(summary1)
                              ↓ previous_context=summary1
chunk2 → LLM → event2(summary2)
                              ↓ previous_context=summary2
chunk3 → LLM → event3

并行模式（速度优先）：
chunk1 ─┐
chunk2 ─┼→ run_in_executor × 4 → LLM 并发 → event1/event2/event3/event4
chunk3 ─┤    （return_exceptions=True 先收集再检查，避免幽灵任务）
chunk4 ─┘
```

#### 健壮性

- LLM 结构化抽取失败直接抛出 `ExtractionError`，**不再有 `_fallback` 静默降级**；
- 单 chunk 抽取失败会向上传播，终止入库，不写入低质量数据；
- 瞬时故障（rate limit/网络抖动）由 `extract_max_retries=2` 自动重试，耗尽后抛错；
- 并行模式用 `return_exceptions=True` 先收集所有结果再统一检查，避免已提交任务成为"幽灵任务"。

产物：`list[ExtractedEvent]`（与 chunks 严格 1:1）

代码位置：[extractor/event_extractor.py](file:///d:/ai_sag_git/ai_sag/extractor/event_extractor.py)、[pipeline.py:101-118](file:///d:/ai_sag_git/ai_sag/ingest/pipeline.py#L101-L118) `_extract_events`、[prompts.py](file:///d:/ai_sag_git/ai_sag/base/prompts.py) `extract_system_prompt`

### 阶段 6：MySQL 事务写入

`db.persist_source(...)` 在**单个事务**内完成所有结构化数据写入，保证原子性。

**写入顺序**：
1. **前置校验**：`len(chunks) != len(events)` 直接抛异常（保证 1:1）；
2. `aisag_sources`：`INSERT ... ON DUPLICATE KEY UPDATE`，`metadata.vector_synced=False`（标记未完成向量同步）；
3. `aisag_documents`：插入 document 记录（存完整原文）；
4. `aisag_chunks`：`executemany` 批量插入所有 chunks；
5. `aisag_events`：`executemany` 批量插入所有 events（每个 event 绑定 chunk_id）；
6. `aisag_entities` + `aisag_event_entities`（逐条处理，需跨 chunk 去重）：
   - 实体去重 key：`(entity_type, normalized_name)`；
   - `_normalize(name) = re.sub(r"\s+", "", name).strip().lower()`（去空白+小写）；
   - 先在本次 `seen_entities` 字典查，再 `INSERT ... ON DUPLICATE KEY UPDATE` + `SELECT id` 查回主键；
   - **`aisag_event_entities.description` 存的是实体在该事件中的 `role`**（供精排阶段使用）；
   - `INSERT IGNORE` 避免重复关联。

**关键设计**：
- 实体**跨 source 全局共享去重**（唯一索引），保证"众商贷"在 A/B 文档中是同一实体节点，图扩展可跨文档；
- `vector_synced=False` 作为"未完成"标记，任何时刻崩溃都可通过对账任务清理；
- 所有写操作在单事务内，失败则全部回滚。

代码位置：[mysql_store.py](file:///d:/ai_sag_git/ai_sag/storage/mysql_store.py) `persist_source`

### 阶段 7：向量化 + Chroma 写入

MySQL 事务成功后，开始向量化并写入 Chroma（共 **5 套向量**，支撑检索不同阶段）：

| Collection | 文本来源 | 用途（检索阶段） | 按 source_id 过滤 |
|------------|----------|------------------|------------------|
| `chunks` | chunk.content | 最终片段向量召回（双路融合 Chunk B + fallback） | ✅ |
| `event_titles` | event.title | 事件标题向量召回（Path B 子路） | ✅ |
| `event_summaries` | event.summary | 事件摘要向量召回（Path B 子路，受 `embed_summary` 开关控制） | ✅ |
| `event_contents` | event.content | 事件粗排余弦相似度 | ✅ |
| `entities` | entity.name | 实体向量扩展召回（Path A） | ❌ 跨 source 共享 |

**写入顺序**：
1. **批量编码 chunks**：`embedder.aembed_texts(chunk_texts)` → `aadd_chunks`；
2. **批量编码事件标题**：`aembed_texts(title_texts)` → `aadd_events(collection="event_titles")`；
3. **批量编码事件内容**：`aembed_texts(content_texts)` → `aadd_event_contents`；
4. **批量编码事件摘要**（`embed_summary=true` 时）：`aembed_texts(summary_texts)` → `aadd_event_summaries`；
5. **实体名批量编码**：
   - 用 `name_lookup` 从 ExtractedEvent 中取原始 name（避免用 normalized 丢中文语义）；
   - 去重后批量编码 → `aadd_entities`（不带 source_id，跨文档共享）；
6. 全部成功 → `db.mark_vector_synced(source_id, True)`。

所有向量写入通过 `asyncio.to_thread` 包装同步的 LlamaIndex/Chroma 调用，保持外层异步接口一致。

代码位置：[pipeline.py:184-258](file:///d:/ai_sag_git/ai_sag/ingest/pipeline.py#L184-L258) `_persist`

### 阶段 8：失败回滚

向量写入过程中任何一步失败：

```
向量写入异常
   ↓
catch vec_err
   ├─ db.delete_by_source(source_id)              → MySQL 事务删除（返回孤儿实体 id）
   ├─ vectors.adelete_by_source(source_id)        → 删 3 个事件/chunk collection 的向量
   └─ vectors.adelete_entities_by_ids(orphan_ids) → 删本次新增的孤儿实体向量
   ↓
重新 raise，API 层返回 500
```

回滚顺序：**MySQL 先删 → 向量库后删**；MySQL 删完返回 `orphan_entity_ids`（只在该 source 出现、删完不再被任何事件引用的实体），一并清理向量。

代码位置：[pipeline.py:259-273](file:///d:/ai_sag_git/ai_sag/ingest/pipeline.py#L259-L273)

## 三、字段抽取详解

### 3.1 内容字段总览

| 表 | 字段 | 类型 | 约束 | 抽取者 | 是否入向量库 |
|----|------|------|------|--------|-------------|
| `aisag_documents` | `content` | MEDIUMTEXT | **可空** | 文件加载器 | ❌ 不入 |
| `aisag_chunks` | `content` | MEDIUMTEXT | **NOT NULL** | 切分器 | ✅ `chunks` 集合 |
| `aisag_events` | `title` | VARCHAR(1024) | NOT NULL | LLM 抽取 | ✅ `event_titles` 集合 |
| `aisag_events` | `summary` | TEXT | 可空 | LLM 抽取 | ✅ `event_summaries` 集合（受开关） |
| `aisag_events` | `content` | MEDIUMTEXT | NOT NULL | LLM 抽取 | ✅ `event_contents` 集合 |

**关键认知**：`events.content` ≠ `chunks.content`，前者是 LLM 重新组织的精炼叙事，后者是切分器产出的原文片段。

### 3.2 内容字段抽取逻辑

#### `aisag_documents.content` —— 文档完整原文
- **抽取者**：文件加载器（`doc_parser/` 下的 pdf/docx/md/xlsx 解析器），无 LLM 参与；
- **规则**：保留文件完整原文（含格式、表格、标题层级），markdown 原样存储，pdf/docx 转纯文本；
- **大小**：与原始文件相当（~50KB），**可空**（URL/流式入库时可仅保留元数据）；
- **作用**：文档详情 API / 下载 API / 全文检索 API（LIKE 匹配），**不参与任何检索/生成环节**，是用户级"原文备份"。

#### `aisag_chunks.content` —— 切片原文
- **抽取者**：切分器（`splitter/chunk_splitter.py`），无 LLM 参与；
- **大小**：约 400-8192 字符/片（受 `chunk_size` 和切分模式影响），50KB 文档约产生数个 chunks；
- **特点**：保留原文格式的**物理切片**，无语义加工，必填；
- **作用**：
  - 检索环节 8（切片回取）：精排后通过 `event.chunk_id` 反查，**用 chunk.content 作为最终喂 LLM 的片段**；
  - 检索环节 9（双路融合 Path B）：query 向量在 `chunks` collection 召回，作为基线片段。

#### `aisag_events.title` —— 事件标题
- **抽取者**：LLM（`structured_predict`）；
- **Prompt**："全局扫描：先识别时间、地点、主体、动作、对象、数据、评价、因果、对比、关系，再写事件""强制单事件融合"；
- **大小**：约 10-25 字（受 `title_max_chars=100` 约束），VARCHAR(1024)；
- **作用**：检索环节 4（标题召回），编码向量存入 `event_titles`，作为 Path B 事件路召回入口。

#### `aisag_events.summary` —— 事件摘要
- **抽取者**：LLM（同 title 一次调用生成）；
- **Prompt**："summary 应为不超过 500 字的压缩摘要，用于快速判断事件相关性"（受 `summary_max_chars=500` 约束）；
- **大小**：≤ 500 字，TEXT 类型；
- **作用**：
  - 检索环节 7（LLM 精排）：`title + summary + entity_roles` 作为候选描述送 LLM 判断相关性（用 summary 因其精炼）；
  - 检索环节 4（摘要召回，受 `embed_summary` 开关）：编码向量存入 `event_summaries`，作为 Path B 子路；
  - 入库抽取时作为下一 chunk 的 `previous_context`，消解代词。

#### `aisag_events.content` —— 事件叙事主线
- **抽取者**：LLM（`structured_predict` 返回 `_Event.content`）；
- **Prompt**："忠实性：不杜撰、不遗漏核心事实、不改主语、不长段照抄原文""content 应为有机叙事主线，非要点列表"；
- **关键约束**：LLM **重新组织**，非复制原文；写成精炼连贯叙述，非列表；
- **大小**：约 100-200 字（比 chunk.content 短一半以上），MEDIUMTEXT NOT NULL；
- **作用**：检索环节 6（事件粗排），编码向量存入 `event_contents`，与 query 向量算余弦相似度排序；
- **关键澄清**：粗排用 `events.content` 编码的**向量**不读原文；最终返回 LLM 的是 `chunks.content` 不是 `events.content`。

### 3.3 实体字段总览

#### `aisag_entities` 表

| 字段 | 类型 | 约束 | 抽取者 | 是否入向量库 |
|------|------|------|--------|-------------|
| `name` | VARCHAR(512) | NOT NULL | LLM 抽取 | ✅ `entities` 集合 |
| `normalized_name` | VARCHAR(512) | NOT NULL | 代码归一化 | ❌ 仅 MySQL（去重键） |
| `description` | TEXT | 可空 | LLM 抽取 | ❌ 仅 MySQL |

唯一键：`UNIQUE KEY uq_aisag_entity (entity_type, normalized_name)` —— 实体跨文档全局共享去重。

#### `aisag_event_entities` 表（事件↔实体 M:N 关联表）

| 字段 | 类型 | 约束 | 抽取者 | 实际存储内容 |
|------|------|------|--------|------------|
| `event_id` | VARCHAR(36) | NOT NULL | 代码关联 | 事件 ID |
| `entity_id` | VARCHAR(36) | NOT NULL | 代码关联 | 实体 ID |

### 3.4 实体字段抽取逻辑

#### `aisag_entities.name` —— 实体原始名称
- **抽取者**：LLM（`structured_predict` 返回 `_Entity.name`）；
- **Prompt**："实体仅作为索引和扩展点，不承载完整语义""并列实体（如'A 和 B'）拆开"；
- **大小**：约 2-30 字；
- **作用**：
  - 检索环节 3（实体向量召回）：编码向量存入 `entities`，召回 Top K 相似实体；
  - 检索环节 7（LLM 精排）：通过 `get_entity_names_by_ids` 查名，格式化为 `"华为(贷款方)"` 喂 LLM。

#### `aisag_entities.normalized_name` —— 归一化名称（去重键）
- **抽取者**：代码归一化（[mysql_store.py](file:///d:/ai_sag_git/ai_sag/storage/mysql_store.py) `_normalize`）：
  ```python
  @staticmethod
  def _normalize(name: str) -> str:
      return re.sub(r"\s+", "", name or "").strip().lower()
  ```
- **归一化规则**：去所有空白 → 去首尾空格 → 转小写（`"华为 技术 有限公司"` → `"华为技术有限公司"`）；
- **作用**：
  - 入库时跨文档全局去重唯一键；
  - 检索环节 3（实体精确匹配）：query 抽取的实体名归一化后用 `normalized_name IN (...)` 精确匹配；
  - 是多跳扩展可跨文档的基础。

#### `aisag_entities.description` —— 实体固有属性
- **抽取者**：LLM（`structured_predict` 返回 `_Entity.description`）；
- **Prompt**："每个 entity.description 必须是该实体的**固有属性**（如'互联网银行'、'科技公司'），**不随事件变化**，建议不超过 20 字"；
- **大小**：≤ 20 字；
- **作用**：⚠️ **当前检索流程实际未读取此字段**（`search_entities_by_name` 查了但仅取 id），属于"设计了但未充分利用"的备用字段。

#### `aisag_event_entities.description`（实际存 role）—— 角色标签
- **抽取者**：LLM 返回 `_Entity.role`，被存入 `event_entities.description` 字段；
- **Prompt**："每个 entity.role 必须是该实体在当前事件中扮演的**角色标签**，4-10 字短词"：
  - organization/person：贷款方/担保方/被投资方/签约主体/被监管方
  - location：事件发生地/注册地/约定履行地
  - time：签约时间/生效时间/违约起始日
  - product：标的物/被担保物/付款对象
  - metric：被引用指标/计算依据/限定条件
  - action：主语行为/约定行为
  - group：受益方/被约束方
  - topic/work/label：主题/被引作品/分类标签
- **大小**：4-10 字；
- **作用**：
  - 检索环节 5（多跳扩展）：仅用 `event_id` 和 `entity_id` 做关联，**不读 description**；
  - 检索环节 7（LLM 精排，**核心作用**）：通过 `_entity_roles_of_events` 读取作为 role，在 `_format_entity_roles` 中格式化为 `"华为(贷款方)"`、`"2024年(签约时间)"`，帮 LLM 理解事件中实体角色判断相关性。

### 3.6 双语义层设计

```
原文层（chunks）
  ├─ content = 切分器原文（chunk_size 字符）
  ├─ 向量 = "原文级"语义
  └─ 用途：Path B 召回、最终回取喂 LLM
                ↓ 不同语义
事件层（events）
  ├─ content = LLM 精炼叙事（100-200 字）
  ├─ 向量 = "事件级"语义
  └─ 用途：Path A 粗排
```

| 向量来源 | 语义类型 | 检索特性 |
|---------|---------|---------|
| `chunks` 向量 | 原文细节语义 | 召回原文细节匹配的片段 |
| `event_contents` 向量 | 事件主旨语义 | 召回事件主题相关的片段 |

两路用不同语义的向量，互补召回，这是双路融合的真正设计意图——不是冗余，是**两层语义互补**。

## 五、跨库一致性保障

### 5.1 核心机制：vector_synced 标记 + 定时对账

```
入库流程：
  MySQL 写入（synced=False）→ 向量库写入 → 标记 synced=True
  ├─ 正常完成：synced=True，数据一致
  ├─ 向量库写失败：补偿删除 MySQL，抛异常
  └─ 进程崩溃（MySQL 已提交，向量库未写完）：synced=False 残留

对账任务（每 300 秒）：
  发现 synced=False 的 source / 孤儿向量 / 孤儿实体 / 软删事件 → 清理
```

### 5.2 对账四类检查

| 类型 | 检查方式 | 覆盖场景 |
|------|----------|----------|
| 第一类 | MySQL 中 `vector_synced=False` 的 source | 入库崩溃、删除中崩溃 |
| 第二类 | 向量库有 source_id 但 MySQL 无 | 删除时 source 物理删但向量库删失败 |
| 第三类 | 向量库有 entity_id 但 MySQL 无 | 孤儿实体向量残留 |
| 第四类 | MySQL 中 `deleted_at IS NOT NULL` 的软删事件 | 软删除事件的物理清理（硬删 MySQL + 向量库） |

对账启动时立即执行一次，之后每 `reconcile_interval` 秒执行一次；任何一类清理失败不影响其它类，日志 warning 后下次重试。设 `AISAG_RECONCILE_INTERVAL=0` 可禁用定时对账（仅启动时对一次）。

代码位置：[pipeline.py:308-389](file:///d:/ai_sag_git/ai_sag/ingest/pipeline.py#L308-L389) `_reconcile`

## 六、删除流程

### 6.1 删除顺序：先向量库后 MySQL（零空洞反向设计）

```
delete_source(source_id)
  │
  ├─ 步骤1：标记删除中
  │   db.mark_vector_synced(source_id, False)
  │
  ├─ 步骤2：向量库删除（检索立即不可见）
  │   vectors.adelete_by_source(source_id)
  │   → 删 chunks / event_titles / event_summaries / event_contents
  │   → 失败不中断，靠对账兜底
  │
  ├─ 步骤3：MySQL 事务删除（页面才消失）
  │   db.delete_by_source(source_id)
  │   → event_entities (DELETE)
  │   → events (UPDATE deleted_at，软删除保留审计)
  │   → chunks (DELETE)
  │   → documents (DELETE)
  │   → sources (DELETE)
  │   → 孤儿 entities (DELETE，NOT EXISTS 引用)
  │   → 返回孤儿实体 id 列表
  │
  └─ 步骤4：孤儿实体向量删除
      vectors.adelete_entities_by_ids(orphan_entity_ids)
      → 失败不中断，靠对账第三类兜底
```

### 6.2 用户体验

| 时间点 | 检索（依赖向量库） | 页面（依赖 MySQL） |
|--------|-------------------|-------------------|
| 删除前 | ✅ 能搜到 | ✅ 能看到 |
| 步骤2后 | ❌ 搜不到了 | ✅ 还能看到 |
| 步骤3后 | ❌ 搜不到 | ❌ 看不到了 |

### 6.3 重建文档（先入新后删旧）

```
rebuild_document(source_id)
  ├─ 1. 用新 source_id 入库（旧数据不受影响）
  ├─ 2. 入库成功后删除旧数据
  └─ 3. 入库失败时旧数据仍在，不影响使用
```

代码位置：[pipeline.py:275-305](file:///d:/ai_sag_git/ai_sag/ingest/pipeline.py#L275-L305) `delete_source`

## 八、异步架构

```
api.py (FastAPI async 端点)
  │  await
  └─→ IngestPipeline (async)
        ├─→ MysqlStore.* (async, aiomysql 连接池)        ← 原生异步
        ├─→ embedder.aembed_texts()                       ← 异步接口（to_thread 包装）
        ├─→ vectors.aadd_chunks() / aadd_events()         ← 异步接口（to_thread 包装）
        └─→ extractor.extract_batch()                     ← to_thread 包装（含 ThreadPoolExecutor）
```

| 组件 | 异步方式 | 原因 |
|------|----------|------|
| MySQL | aiomysql 原生异步 | 网络 IO |
| Embedder | `asyncio.to_thread` 包装 | CPU 密集（本地模型推理） |
| VectorStore | `asyncio.to_thread` 包装 | 本地磁盘 IO（ChromaDB 无异步 API） |
| LLM（入库抽取） | `asyncio.to_thread` / `run_in_executor` 包装 | LlamaIndex 同步 + 内部 ThreadPoolExecutor |

**并发控制**：
- 入库级：`_ingest_semaphore = Semaphore(concurrency=2)`，限制同时入库文档数；
- 文档内：并行抽取模式 `run_in_executor` + `asyncio.gather(return_exceptions=True)`，workers=4。

## 九、核心设计决策

1. **统一 11 类实体口径（入库 = 查询）**
   - `ENTITY_TYPES / ENTITY_NAME_NORM / entity_types_detail()` 都定义在 `base/prompts.py`；
   - 查询侧 `query_extract_system_prompt()` 复用同一套规范，从根本上避免"入库抽的是'华为技术有限公司'、查询抽的是'华为'"导致对不上的问题。

2. **单事件融合原则**
   - 每个 chunk 只抽一个综合顶层事件，不拆三元组；
   - 事件 `content` 是有机叙事主线而非要点列表，后续作为 `event_contents` 向量文本语义完整。

3. **description vs role 双字段设计**
   - `entities.description`：实体**固有属性**，全局不变；
   - `event_entities.description`（关联表，字段名误导）：实体**在该事件中的角色**，随事件变化；
   - 查询精排阶段把"众商贷(贷款方)"这类角色信息渲染给 LLM，解决方向性问题（"谁贷给谁"）。

4. **实体跨 source 全局去重共享**
   - 唯一键 `(entity_type, normalized_name)`，归一化规则：去空白 + 小写；
   - 向量实体 collection 不按 source_id 过滤（任何文档中的"众商贷"都是同一个向量节点）；
   - 删文档时必须检查实体是否还被其它事件引用，只删真正的孤儿。

5. **chunks 与 events 严格 1:1**
   - 每个 chunk 对应一个事件，每个事件绑定一个 chunk_id；
   - `persist_source` 入口强校验，避免长度不匹配导致错位；
   - LLM 抽取失败直接抛出 `ExtractionError`，上层负责重试或记录。

6. **写入顺序：先 MySQL 后向量，失败双向回滚**
   - MySQL 先写（带 `vector_synced=False` 标记）→ 再写向量 → 最后标记 True；
   - 任意一步失败都清理两边残留，结合定时对账兜底，最终一致。

7. **删除顺序：先向量后 MySQL（零空洞反向设计）**
   - 标记 `vector_synced=False` → 先删向量库（检索立即查不到）→ 再删 MySQL（页面稍后消失）；
   - 避免"页面显示已删除但检索还能搜到"的尴尬窗口。

8. **重建策略：先入新后删旧**
   - `PUT /api/documents/{source_id}` 用新 source_id 入库成功后再删旧 source_id；
   - 入库失败旧数据完全不受影响，避免"先删后入失败导致文档消失"。

9. **semantic 切分为默认**
   - semantic 切出的 chunk 按语义断点划分，天然保持语义完整，LLM 抽取事件质量最高；
   - 代价：入库时需要额外的 embedding 计算（复用已有 embedder），速度比 sentence 慢，但为一次性成本。

10. **文体感知抽取（方案 A+B）**
    - 文档级文体识别（每文档 1 次 LLM 调用）驱动抽取规则和 role 词汇表；
    - 短路优化（xlsx/csv → tabular）省调用；
    - 失败降级 generic 不阻断入库；查询侧不调用保证口径一致。

11. **Excel 解析容错（v2 版本）**
    - 四层防御校验（存在/扩展名/大小/ZIP 签名）；
    - `_load_workbook_safe` 对样式表损坏的非标准 xlsx，替换为最小有效 styles.xml（预填充 1000 个 `<xf/>`），覆盖 99% 实际损坏场景。

12. **并发入库信号量**
    - `_ingest_semaphore = Semaphore(concurrency=2)` 限制同时入库文档数；
    - 防 LLM API rate limit 和 embedding OOM，可按机器配置调整。

## 十、数据流全景图

### 10.1 入库阶段

```
原始文档
    │
    ├─→ documents.content    (加载器，完整原文)
    │       └─ 用于：文档详情、下载、全文检索（不参与问答）
    │
    ├─→ chunks.content       (切分器，chunk_size 字/片)
    │       ├─ 用于：向量编码 → Chroma chunks 集合
    │       ├─ 用于：精排后回取原文
    │       └─ 用于：Path B 向量召回
    │
    └─→ events.title/summary/content   (LLM 抽取)
            ├─ title   → 向量编码 → Chroma event_titles
            ├─ summary → 向量编码 → Chroma event_summaries（受开关）
            │          └─ 仅 MySQL 也可用于精排
            └─ content → 向量编码 → Chroma event_contents

实体（LLM 抽取）
    ├─→ entities.name             (原始名称)
    │       └─ 用于：精排展示 "华为(贷款方)" / 向量召回
    ├─→ entities.normalized_name   (代码归一化)
    │       ├─ 用于：跨文档去重唯一键
    │       └─ 用于：实体召回精确匹配
    ├─→ entities.description      (LLM 抽取的固有属性)
    │       └─ ⚠️ 当前流程未使用
    ├─→ Chroma entities.text = name     (向量编码源)
    │       └─ 用于：实体向量扩展召回
    └─→ event_entities.description = role   (⚠️ 字段名误导)
            └─ 用于：精排时格式化 "华为(贷款方)"
```

### 10.2 与检索阶段的衔接

入库产出的 5 套向量 + MySQL 结构化数据，支撑检索阶段（详见 [检索流程.md](file:///d:/ai_sag_git/ai_sag/docs/检索流程.md)）：

| 入库产物 | 检索环节 | 作用 |
|---------|---------|------|
| `chunks` 向量 | 双路融合 Path B / fallback | 基线片段召回 |
| `event_titles` 向量 | Path B 标题召回 | 事件标题向量召回 |
| `event_summaries` 向量 | Path B 摘要召回 | 事件摘要向量召回 |
| `event_contents` 向量 | 事件粗排 | 余弦相似度排序 |
| `entities` 向量 | Path A 实体扩展 | 实体向量近邻召回 |
| `event_entities` 关联表 | BFS 多跳扩展 | 事件↔实体关联遍历 |
| `event_entities.description`(role) | LLM 精排 | 角色标注 `"华为(贷款方)"` |
| `chunks.content` 原文 | 切片回取 | 最终喂 LLM 的片段 |

## 十一、核心结论

### 11.1 各字段定位

| 字段 | 定位 | 是否可删 |
|------|------|---------|
| `documents.content` | 用户级备份（下载/展示） | ⚠️ 可空，删后影响下载 |
| `chunks.content` | **检索核心**（喂 LLM 的原文） | ❌ 删后问答瘫痪 |
| `events.title` | 标题召回入口 | ❌ 删后标题召回失效 |
| `events.summary` | LLM 精排候选描述 + 摘要召回 + 跨片段上下文 | ❌ 删后精排/召回/消解失效 |
| `events.content` | 事件级粗排向量源 | ❌ 删后丢失事件级语义 |
| `entities.name` | 实体标识，向量编码源，精排展示 | ❌ 删后实体召回失效 |
| `entities.normalized_name` | 跨文档去重键，精确匹配 | ❌ 删后实体无法去重和精确匹配 |
| `entities.description` | 实体固有属性 | ⚠️ 当前未使用，可暂留备用 |
| `event_entities.description`(role) | 精排时角色展示 | ❌ 删后精排丢失角色信息 |
| `event_entities.weight` | 实体权重 | ⚠️ 固定 1.0，当前未参与计算 |

### 11.2 关键认知澄清

| 易混淆点 | 正确事实 |
|---------|---------|
| `events.content` = `chunks.content`？ | ❌ events.content 是 LLM 精炼的叙事主线 |
| `events.content` 是冗余设计？ | ❌ 是双语义层设计（原文级 + 事件级） |
| 双路融合是冗余？ | ❌ 是两层语义互补 |
| `event_entities.description` 存的是什么？ | **存的是 `entity.role`（角色），不是 `entity.description`（固有属性）** |
| `entities.description` 和 `event_entities.description` 区别？ | 前者是固有属性（不随事件变化），后者是角色（随事件变化） |
| `entities.description` 在检索中使用吗？ | ❌ 当前流程未读取 |
| `event_entities.weight` 有作用吗？ | ⚠️ 固定 1.0，当前未参与计算 |

### 11.3 一句话总结

> **`documents.content` 是"备份"，`chunks.content` 是"检索核心"，`events.title/summary/content` 是"事件路三级语义"，`entities.name/normalized_name` 是"实体召回与去重键"，`event_entities.description`(role) 是"精排角色标注"**——各司其职，共同构成 ai_sag 的双语义层 + 实体路检索闭环。
