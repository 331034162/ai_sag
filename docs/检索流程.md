## 一、整体架构

ai_sag 检索遵循 SAG 论文（SQL-Retrieval Augmented Generation with Query-Time Dynamic Hyperedges）思路：**离线零结构，查询时用 SQL Join 动态织网，用完即弃**。

核心是**双路种子召回 + BFS 动态超边扩展 + 粗排 + LLM 精排 + 双路融合**：

```
用户 Query
   │
   ▼
┌─────────────────────────────┐
│  0. 查询重写（多轮时）        │  ← 消解代词，产出独立可检索的 eff_query
└─────────────┬───────────────┘
              │ eff_query
              ▼
┌─────────────────────────────┐
│  1. Query Embedding         │
└─────────────┬───────────────┘
              │ query_vec
       ┌──────┴──────┐ 并发
       ▼             ▼
┌──────────────┐ ┌──────────────────┐
│ Path A       │ │ Path B           │
│ LLM抽实体    │ │ title+summary    │
│ 精确+向量扩展 │ │ 向量召回         │
│ 度数过滤     │ │ similarity>0.4   │
│ SQL反查事件  │ │                  │
└──────┬───────┘ └────────┬─────────┘
       │                  │
       ▼                  ▼
┌─────────────────────────────┐
│  4. 合并种子事件池（去重）    │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────────────────┐
│  5. BFS 多跳扩展（hopllm 动态停止）       │
│     事件→实体→度数过滤→综合评分→新事件    │
│     ← SQL 动态超边，用完即弃 →            │
└─────────────┬───────────────────────────┘
              │
              ▼
┌─────────────────────────────┐
│  6. 粗排（event_contents 余弦）│  ← Top max_events(100)
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  7. LLM 精排（实体信号+深度衰减）│  ← Top rerank_top_k(5)
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  8. 切片回取 ChunkA          │  ← 事件→chunk 原文
└─────────────┬───────────────┘
              │
   ┌──────────┴──────────┐
   ▼                     ▼
┌──────────┐    ┌──────────────────┐
│ ChunkA   │    │ 9. 基线向量 ChunkB │  ← chunks 向量召回
└────┬─────┘    └────────┬─────────┘
     │                   │
     ▼                   ▼
┌─────────────────────────────┐
│  10. 双路融合（concat/supplement）│
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  11. LLM 答案生成            │
└─────────────────────────────┘
```

**接口入口**：

| 接口 | 路径 | 用途 | 引擎方法 |
|------|------|------|----------|
| 检索 | `POST /api/search` | 仅检索不回答 | `QAEngine.search` |
| 单轮问答 | `POST /api/ask` | 一问一答 | `QAEngine.ask` |
| 多轮对话 | `POST /api/chat` | 带历史完整回答 | `QAEngine.chat` |
| 流式对话 | `POST /api/chat/stream` | SSE 流式输出 | `QAEngine.chat_stream` |

## 二、检索全流程详解

### 2.0 查询重写（Query Rewrite）

- **触发条件**：仅多轮对话（`history` 非空），单轮问答跳过。
- **作用**：消解代词（"它/这家公司/那个产品"）和省略主语，把依赖上文的追问重写为独立、完整、可直接检索的查询。
- **实现**：取最近 `rewrite_max_rounds=5` 轮历史，格式化为 `用户：xxx\n助手：xxx`，调用 LLM（`achat`）重写；失败回退原 query。
- **产物**：`eff_query`，后续所有检索步骤均使用 `eff_query`。

代码位置：[sag_retriever.py](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py)

### 2.1 向量编码

- 对 `eff_query` 调用 `embedder.aembed_text()` 生成 `query_vec`。
- 与 Path A 实体抽取、Path B 标题召回**并发执行**（`asyncio.gather`）。

### 2.2 Path A：结构化实体召回

Path A 通过"实体 → SQL 关联"召回事件，解决多跳关联问题。

#### 2.2.1 查询实体抽取（LLM）

- 用 `astructured_predict` 让 LLM 输出查询实体名列表；
- 覆盖 11 类实体（time/location/person/organization/group/topic/work/product/action/metric/label），与入库抽取共用同一份规范，保证口径一致；
- 失败降级为正则分词（当前返回空列表）。

代码位置：[sag_retriever.py:305](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py#L305) `_extract_query_entities`

#### 2.2.2 实体精确匹配 + 向量扩展

召回分两步，受 `entity_expand_enabled` 开关控制：

| 步骤 | 机制 | 开关 |
|------|------|------|
| 精确匹配 | `db.search_entities_by_name(names)` SQL 按名查 entity_id | 始终执行 |
| 向量扩展 | 实体名 embedding 去向量库找近邻，补充同义词/相关实体 | `entity_expand_enabled=true` 时执行 |

**向量扩展关键参数**：
- `entity_expand_threshold=0.5`：相似度阈值。基于三个对照 query 验证：对"信贷系统"保留 6 个同义词（0.508~0.894），对"大模型"保留 4 个核心概念实体（0.511~0.574），对"架构团队"挡住绝大部分噪声（最高 0.61）。0.7 会误杀"大模型"类概念扩展，0.3 会引入大量噪声，0.5 是当前数据下的平衡点。
- `entity_expand_topk=20`：每个查询实体名召回的近邻数。
- 并发查询：所有实体名 embedding 并发查询向量库（`asyncio.gather`）。

代码位置：[sag_retriever.py](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py) `_recall_entities`

#### 2.2.3 种子实体度数过滤

向量扩展 + 精确匹配合并去重后，做度数过滤防止高频枢纽实体污染种子池：

1. **总量上限**：`seed_entity_budget=20`，超过截断；
2. **绝对上限**：`entity_degree_abs_max=50`，度数 > 此值的实体直接剔除；
3. **离群检测**：当 batch ≥ `seed_entity_min_batch=15` 时，用 `entity_degree_method`（默认 `otsu`）计算离群阈值，剔除超阈值实体；
4. **过度过滤保护**：若离群后剩余 < `max(len//2, seed_entity_min_keep)`，回退到绝对上限结果，避免断链。

> 与 BFS 边界过滤的区别：种子阶段不用综合评分截断（向量扩展已按相似度筛过），离群检测用更保守的 `min_batch`，保护用更高的 `min_keep`。

代码位置：[sag_retriever.py:356-386](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py#L356-L386)

#### 2.2.4 SQL 反查事件

`db.get_event_ids_by_entity_ids(entity_ids, source_ids)` 联表查关联事件 → **种子事件集 A**。

### 2.3 Path B：事件向量召回

`seed_recall=mixed`（默认）双路并发：

| 子路 | collection | 说明 |
|------|-----------|------|
| title | `event_titles` | 事件标题向量召回 |
| summary | `event_summaries` | 事件摘要向量召回 |

- 两路 `top_k=max_events`，`similarity_threshold=0.4`（论文 Path B）；
- 合并：去重保留 max 分数，按分降序 → **种子事件集 B**。

代码位置：[sag_retriever.py:78-100](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py#L78-L100)

> **关于 `similarity_threshold=0.4`**：句子级语义匹配（query ↔ 事件 title/summary）分数天然低于短语同义匹配（实体名 ↔ 实体名），故 0.4（事件）低于 0.5（实体）是合理的。0.4 在"大模型"query 上保住了 0.4127 的边缘相关事件，在"架构团队"query 上挡住所有噪声（0 召回），是稳的。

### 2.4 合并种子事件池

```python
seed_ids = list(dict.fromkeys(entity_event_ids + title_summary_event_ids))  # 保序去重
```

- Path A 优先（图谱结构化召回更精准）；
- 若 `seed_ids` 为空，走兜底路径 `_fallback()`（纯 chunk 向量召回），不进行图扩展。

### 2.5 BFS 多跳扩展（SAG 核心）

从种子事件出发，沿"事件 ↔ 实体 ↔ 事件"关联边做多跳扩展，召回语义关联的周边事件。详见 [第四节](#四bfs-多跳扩展深度解析)。

代码位置：[sag_retriever.py:431-534](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py#L431-L534) `_expand`

### 2.6 粗排（Coarse Rank）

- **目的**：BFS 扩展后事件可能达数百个，先用向量相似度做廉价筛选。
- **实现**：批量取 `event_contents` 向量，用 numpy 计算与 `query_vec` 的余弦相似度，降序取前 `max_events=100`。
- 余弦公式：`score = dot(q, e) / (||q|| * ||e|| + 1e-8)`，缺失 embedding 得分 0.0。

代码位置：[sag_retriever.py:807](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py#L807) `_coarse_rank`

### 2.7 LLM 精排

#### 候选命中信号计算

精排前先计算每个候选事件的"实体命中信号"，作为 LLM 判断依据：

- **命中检测**：遍历候选事件的 `entity_roles`，若实体在查询实体集中，渲染为 `"华为(贷款方)"` 等带角色标注，并统计命中数 `[命中 2/3]`；
- **信号权重**：按实体在向量扩展中的相似度分档（strong/weak），结合 IDF 加权；
- **深度衰减**：`adjusted = coarse_score × decay^event_depth`，种子事件 depth=0（系数 1.0），BFS 各跳 depth=hop+1（系数 decay、decay²...），抑制深层扩展噪声。

#### LLM 重排

- 输入：`top_n[:rerank_candidate_limit=100]` 候选（含标题+摘要+命中信号）；
- LLM 用 `astructured_predict` 输出按相关性降序的编号列表；
- 失败降级为 `acomplete` + 正则解析，再不行用粗排顺序；
- 取前 `rerank_top_k=5`。

代码位置：[sag_retriever.py:867-898](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py#L867-L898) `_rerank` / `_llm_rerank`

### 2.8 切片回取（Chunk A）

- `db.get_chunk_ids_by_event_ids(reranked_ids)` → chunk_id 映射；
- `db.get_chunks_by_ids(chunk_ids)` → chunk 原文；
- 包装为 `RetrievedSection`，顺序严格遵循精排顺序，`score` 线性衰减（1.0, 0.99, ...）。

### 2.9 基线向量召回（Chunk B）

- `vectors.aquery_chunks(query_vec, top_k=2*max_sections, similarity_threshold=0.4)`；
- 按 `source_ids` 过滤，跳过已存在 chunk；
- 语义基线，兜底图谱未覆盖的场景。

### 2.10 双路融合

| 模式 | 行为 |
|------|------|
| `concat`（默认） | A、B 拼接去重，A 优先 |
| `supplement` | A 为主，不足 `max_sections` 时用 B 补足 |

- 去重用 `chunk_id` 集合；
- 截断为 `max_sections=5`。

代码位置：[sag_retriever.py:243](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py#L243)

### 2.11 答案生成

由 `QAEngine` 组装 prompt 调用 LLM：

- **单轮（ask）**：`[SYSTEM(含资料), USER(query)]`；
- **多轮（chat）**：`[SYSTEM(含资料), ...历史(截断最近5轮), USER(query)]`，历史不重复携带资料避免 token 膨胀；
- **流式（chat_stream）**：先同步返回 `meta` 事件（sections/trace），再 SSE 推流 token，最后 `done`；
- 片段格式：`【片段{i}】(来源: {source}, 小节: {heading})\n{content}`；
- 检索无结果返回 `QA_EMPTY_ANSWER`。

代码位置：[qa_engine.py](file:///d:/ai_sag_git/ai_sag/retrieval/qa_engine.py)

## 三、多轮对话历史的传导机制

历史在检索阶段**只直接作用于查询重写**，产出的 `eff_query` 像一根线把历史信息传导到所有下游环节；在生成阶段**再次直接参与**消息组装。

### 3.1 传导链路

```
history ──┐
query ────┤
          ▼
   0. 查询重写 [LLM]      ← 历史 + query → eff_query（历史唯一直接参与）
          │ eff_query
          │
   1. 向量编码  [计算]     eff_query → query_vec
   2. 实体抽取  [LLM]      eff_query → query_entities
   3. 标题召回  [向量库]    query_vec → 种子事件B
   4. 实体召回  [数据库]    query_entities → 种子事件A
   5. 多跳扩展  [数据库]    seed_ids → expanded_ids（与历史无直接关系）
   6. 粗排     [向量库]    query_vec + expanded_ids → ranked_ids
   7. LLM 精排 [LLM]       eff_query + ranked_ids → reranked_ids
   8. 切片回取 [数据库]    reranked_ids → sections（与历史无直接关系）
   9. 双路融合 [向量库]    query_vec + sections → 最终 sections
  10. 答案生成 [LLM]       ← 历史再次直接参与（作为多轮消息列表）
```

### 3.2 各环节历史参与度

| 环节 | 数据源 | 历史参与 | 失效后果 |
|------|--------|---------|---------|
| 0. 查询重写 | LLM | **直接** | eff_query 错误，全链路失效 |
| 1. 实体抽取 | LLM | 间接（eff_query） | 抽不到正确实体，图谱路径失效 |
| 2. 向量编码 | 计算 | 间接（eff_query） | 向量偏移，召回不准 |
| 3. 标题召回 | 向量库 | 间接（query_vec） | 召回不到相关事件 |
| 4. 实体召回 | 数据库 | 间接（query_entities） | 图谱种子缺失 |
| 5. 多跳扩展 | 数据库 | 无（输入 seed_ids） | 扩展范围由种子决定 |
| 6. 粗排 | 向量库 | 间接（query_vec） | 排序偏移 |
| 7. LLM 精排 | LLM | 间接（eff_query） | LLM 不理解真实意图 |
| 8. 切片回取 | 数据库 | 无（输入 reranked_ids） | 与查询无关 |
| 9. 双路融合 | 向量库 | 间接（query_vec） | 补充切片不相关 |
| 10. 答案生成 | LLM | **直接** | 答案不连贯，代词无法消解 |

### 3.3 两阶段使用对比

| 维度 | 检索阶段（查询重写） | 生成阶段（消息组装） |
|------|---------------------|---------------------|
| 历史格式 | 纯文本拼接 | 原生 ChatMessage 列表 |
| 资料携带 | ❌ 不带 | ✅ 当前轮资料放 SYSTEM |
| 截断轮数 | `rewrite_max_rounds=5` | `MAX_HISTORY_ROUNDS=5` |
| 使用的 query | eff_query（重写后） | 原始 query（未重写） |

**为什么检索用 eff_query，生成用原始 query**：检索系统需要"独立完整的查询"才能精准匹配；生成阶段 LLM 有能力从原始表达 + 历史上下文理解意图，且保留原始表达让答案更自然。

### 3.4 资料不进历史

每轮检索的资料片段**不进入历史**，只保留问答文本。原因：避免 token 爆炸（5 轮 × 5 片段 × 512 字 ≈ 13.8K，逼近上下文上限）。每轮重新检索，当前轮资料放 SYSTEM。

## 四、BFS 多跳扩展深度解析

### 4.1 子策略对比

| 子策略 | 机制 | 特点 |
|--------|------|------|
| `multi` | 固定跳数，每跳新事件经 budget 截断作下一跳种子 | 截断由 SQL `SELECT DISTINCT` 顺序决定（无 ORDER BY），本质随机抽样，**设计缺陷** |
| `hopllm`（默认） | 每跳用向量相似度重排新事件，取 topK 作种子，低于阈值动态停止 | 质量优先，可控性强 |

### 4.2 hopllm 动态停止

每跳发现新事件后：

1. 计算 `query_vec ↔ event_contents` 余弦相似度，降序排列；
2. 取最高分 `best_score`；
3. 若 `best_score < hop_relevance_threshold=0.15` → **停止扩展**（已发现事件保留，交粗排/精排判断）；
4. 否则取前 `hop_seed_topk=8` 作为下一跳种子继续。

> `hop_relevance_threshold=0.15` 是"防御性安全阀"而非"质量门控"。当前 `event_contents` 分数分布为 0.4+，此阈值实际很少触发，主要靠 `entity_expand_threshold` 在源头控制 BFS 规模。设计哲学是"停止信号"而非"过滤阈值"——已 track 的事件不丢弃，交给粗排/精排决定去留。

### 4.3 边界实体度数过滤

BFS 每跳从当前事件取出所有实体后，做两层过滤：

**第一层：度数硬过滤**

1. **绝对上限**：`entity_degree_abs_max=50`，度数 > 此值直接剔除；
2. **离群检测**：当 batch ≥ `entity_degree_min_batch=10` 时，按 `entity_degree_method` 计算阈值：

| 方法 | 公式 | 说明 |
|------|------|------|
| `otsu`（默认） | 大津法自动找最优二分阈值 | 数据驱动，无需手动设分位数 |
| `percentile` | batch 内 P95 分位数 | 自适应 batch 分布 |
| `mad` | median + k×MAD/0.6745 | 对长尾鲁棒，k=3.0 对应 99.73% |
| `tukey` | Q3 + k×IQR | 经典箱线图，k=3.0 较保守 |
| `none` | 关闭 | 仅用绝对上限 |

3. **过度放宽检测**：若硬过滤后样本范围过宽（`relaxed=True`），标记但不阻断，交给综合评分处理。

**第二层：综合评分截断**

当硬过滤后实体数 > `entity_frontier_budget=100` 时，做综合评分取 top budget：

- **IDF 评分**：`1.0 / (1.0 + degree)`，低频实体得分高（抑制枢纽实体）；
- **query 相似度评分**：`query_vec ↔ entity` 余弦相似度（保留与 query 语义相关实体）；
- **归一化**：winsorize + min-max（先按 P10/P90 分位数裁剪极端值再归一化，避免 min-max 边界塌缩）；
- **综合分**：`combined = (1-α) × idf_norm + α × sim_norm`，`α=entity_frontier_query_weight=0.6`；
- 按综合分降序取 top 100。

> **winsorize+min-max 的动机**：纯 min-max 对异常最值敏感——一个离群小值会把其余样本压到接近 1，相对差异被压缩（"边界塌缩"）。winsorize 先裁剪极端值再做归一化，保留主体分布的相对差异。

代码位置：[sag_retriever.py:700-759](file:///d:/ai_sag_git/ai_sag/retrieval/sag_retriever.py#L700-L759) `_filter_frontier_entities`

### 4.4 事件深度语义

BFS 记录每个事件的发现深度，供精排计算衰减后相关度：

| 深度 | 来源 | 衰减系数 |
|------|------|---------|
| depth=0 | 种子事件（实体召回 + 向量召回） | 1.0 |
| depth=1 | BFS hop=0 发现的新事件 | decay |
| depth=2 | BFS hop=1 发现的新事件 | decay² |
| depth=3 | BFS hop=2 发现的新事件 | decay³ |

精排调整后相关度 = 向量相关度 × decay^event_depth，抑制深层扩展噪声。

### 4.5 性能隐患与优化建议

BFS 每跳涉及 3 个核心 SQL：

| SQL | 作用 | 索引命中 |
|-----|------|---------|
| 事件详情 | `aisag_events WHERE id IN` | PRIMARY KEY ✅ |
| 实体角色 | `aisag_event_entities WHERE event_id IN` | `idx_aisag_ee_event` ✅ |
| 实体反查事件 | `aisag_event_entities JOIN aisag_events WHERE entity_id IN` | `idx_aisag_ee_entity` ✅ + PRIMARY KEY ✅ |

**潜在隐患**：

| # | 隐患 | 严重度 | 触发阈值 | 解决方案 |
|---|------|--------|---------|---------|
| 1 | `deleted_at` 无独立索引 | ★★ 中 | 事件 > 5万 | `ADD INDEX idx_aisag_event_deleted (deleted_at)` |
| 2 | JOIN 放大效应 | ★★ 中 | 扩展事件 > 200 | 每跳即时裁剪 TopK（已由 `entity_frontier_budget` 缓解）|
| 3 | `max_events` 截断滞后 | ★★ 中 | 扩展事件 > 200 | `_expand` 内部裁剪 |
| 4 | N+1 实体角色查询 | ★★ 中 | 事件 > 50 | JOIN 合并为单次查询 |
| 5 | `NOT IN` 排除效率 | ★ 低 | tracked > 50 | 改 `LEFT JOIN ... IS NULL` |
| 6 | `SELECT DISTINCT` 临时表 | ★ 低 | 单跳 > 1000 行 | 改 `GROUP BY` 或应用层去重 |
| 7 | 无查询缓存 | ★ 低 | 多轮重复实体 | LRU/TTL 缓存 |

**当前结论**：现有数据规模下性能无感知影响。`entity_frontier_budget=100` 和 `hop_seed_topk=8` 已有效控制每跳规模。待事件数突破 5 万或单轮扩展 > 200 时，启动隐患 1/3/4 优化。

监控 SQL：
```sql

  (SELECT COUNT(*) FROM aisag_events WHERE deleted_at IS NULL) AS active_events,
  (SELECT COUNT(*) FROM aisag_event_entities) AS relations,
  (SELECT AVG(cnt) FROM (SELECT entity_id, COUNT(*) cnt FROM aisag_event_entities GROUP BY entity_id) t) AS avg_events_per_entity;
```

## 六、模块调用关系

```
api.py (FastAPI 路由)
  └─> QAEngine (qa_engine.py)           # 编排层：search/ask/chat/chat_stream
        ├─> SagRetriever (sag_retriever.py)  # 检索核心
        │     ├─> BaseEmbedder               # 文本向量化
        │     ├─> BaseVectorStore            # Chroma 向量 ANN 检索
        │     │     └ collections: chunks / event_titles / event_summaries / event_contents / entities
        │     ├─> MysqlStore                 # aiomysql 异步存储
        │     │     ├─> sources/documents/chunks
        │     │     ├─> events / event_entities(含 role)
        │     │     └─> entities
        │     └─> LLM (llm/)                # 查询重写/实体抽取/精排
        └─> LLM (生成答案/流式输出)
```

### 关键设计决策

1. **全链路异步**：MySQL 用 aiomysql，LLM 用 LlamaIndex 原生异步，Embedder/VectorStore 用 `asyncio.to_thread` 包装；无依赖步骤 `asyncio.gather` 并发。
2. **入库与查询口径一致**：11 类实体类型、命名规范在入库和查询抽取中共用同一份常量，避免口径漂移。
3. **实体角色信号融入精排**：入库阶段为每个 (事件, 实体) 对抽取角色标签，精排时渲染给 LLM，区分方向性问题。
4. **双路融合兜底**：图谱路径擅长结构化关系推理，纯向量路径擅长语义相似兜底；seed 为空时回退纯向量召回。
5. **多层截断与保护**：`rerank_candidate_limit` 防 prompt 过长，`rerank_top_k`/`max_sections` 控制输出，度数过滤防枢纽实体污染，过度过滤保护防断链。
6. **容错降级**：重写失败用原 query，实体抽取失败用空列表，精排失败用粗排顺序，检索无结果返回兜底文案。
