## 四、检索配置（`SearchConfig`）— **核心**

[base/config.py:96-107](file:///d:/ai-llamaindex/ai_sag/base/config.py#L96-L107)

### 完整参数表

| 参数 | 环境变量 | 默认值 | 作用环节 | 说明 |
|------|---------|--------|---------|------|
| `similarity_threshold` | `AISAG_SIMILARITY_THRESHOLD` | `0.4` | 3.实体召回 / 6.粗排 / 基线 chunk 召回 | 相似度阈值，低于此值丢弃（论文 Path B） |
| `entity_expand_threshold` | `AISAG_ENTITY_EXPAND_THRESHOLD` | `0.9` | 3.实体召回（Path A 扩展） | 实体向量扩展阈值，严格过滤噪声实体 |
| `max_hops` | `AISAG_MAX_HOPS` | `2` | 5.多跳 BFS 扩展 | 多跳扩展最大跳数（论文 H=1，ai_sag 默认 2） |
| `max_events` | `AISAG_MAX_EVENTS` | `100` | 6.粗排后截断 | 粗排后保留的最大事件数 |
| `rerank_top_k` | `AISAG_RERANK_TOP_K` | `5` | 7.LLM 精排 | 精排后保留的 TopK 事件数 |
| `rerank_candidate_limit` | `AISAG_RERANK_CANDIDATE_LIMIT` | `100` | 7.LLM 精排 | 送入 LLM 精排的最大候选数（防 prompt 过长） |
| `max_sections` | `AISAG_MAX_SECTIONS` | `5` | 8.切片回取 | 最终返回给 LLM 的片段数 |
| `fusion` | `AISAG_FUSION` | `concat` | 9.双路融合 | 融合策略：`concat`（拼接）/ `supplement`（补充） |
| `entity_frontier_budget` | `AISAG_ENTITY_FRONTIER_BUDGET` | `100` | 5.多跳 BFS 扩展 | BFS 每跳边界实体数量上限（论文 Section 4.4） |
| `sub_strategy` | `AISAG_SUB_STRATEGY` | `hopllm` | 5.多跳 BFS 扩展 | 子策略：`multi`（固定跳数）/ `hopllm`（每跳相似度动态停止） |
| `hop_relevance_threshold` | `AISAG_HOP_RELEVANCE_THRESHOLD` | `0.15` | 5.多跳 BFS 扩展 | hopllm 动态停止阈值 |
| `hop_seed_topk` | `AISAG_HOP_SEED_TOPK` | `8` | 5.多跳 BFS 扩展 | hopllm 每跳重排后保留的种子事件数 |
| `rewrite_max_rounds` | `AISAG_REWRITE_MAX_ROUNDS` | `5` | 1.查询重写 | 查询重写阶段保留的历史轮数 |

### 参数在 9 环节流程中的位置

```
1.查询重写      [LLM]      ← LlmConfig
2.实体抽取      [LLM]      ← LlmConfig
3.实体召回      [向量库]   ← similarity_threshold=0.4, entity_expand_threshold=0.9
4.标题召回      [向量库]   ← similarity_threshold=0.4
5.多跳扩展      [数据库]   ← max_hops=2
6.粗排          [向量库]   ← max_events=100, similarity_threshold=0.4
7.LLM精排       [LLM]     ← rerank_candidate_limit=100, rerank_top_k=5
8.切片回取      [数据库]   ← max_sections=5
9.双路融合      [计算]     ← fusion=concat
10.答案生成     [LLM]     ← LlmConfig + MAX_HISTORY_ROUNDS=5
```

### 参数调整建议

| 场景 | 调整建议 |
|------|---------|
| 召回太少，答非所问 | `max_hops` 升到 3，`similarity_threshold` 降到 0.3 |
| 召回太多噪声，答案跑题 | `max_hops` 降到 1，`similarity_threshold` 升到 0.5 |
| 精排太慢 | `rerank_candidate_limit` 降到 10 |
| 答案片段不够 | `max_sections` 升到 8 |
| 想要补充信息 | 前端传 `fusion=supplement` |

### 历史截断逻辑

[qa_engine.py:118](file:///d:/ai-llamaindex/ai_sag/retrieval/qa_engine.py#L118)

```python
truncated = self._truncate_history(history, MAX_HISTORY_ROUNDS)
```

**说明**：
- `MAX_HISTORY_ROUNDS=5` 表示保留最近 5 轮（5 个 user + 5 个 assistant = 10 条消息）
- 超过的旧历史被丢弃，避免 prompt 超长
- 修改后需调整 LLM 的 `context window` 容量评估

### 历史参与方式

| 环节 | 历史参与方式 | 说明 |
|------|------------|------|
| 1.查询重写 | **直接** | 读取 history 生成 eff_query |
| 2.实体抽取 | 间接 | 用 eff_query 衍生物 |
| 3-9.检索各环节 | 间接/无 | 通过 eff_query 传导 |
| 10.答案生成 | **直接** | history 作为多轮消息送入 LLM |

## 九、参数调优速查表

### 9.1 按问题现象调参

| 现象 | 调整参数 | 推荐值 | 原因 |
|------|---------|--------|------|
| 答非所问 | `similarity_threshold` | 0.3 | 召回阈值过高，放宽 |
| 答案跑题 | `similarity_threshold` | 0.5 | 召回噪声多，收紧 |
| 找不到关联信息 | `max_hops` | 3 | 多跳扩展不够 |
| 精排太慢 | `rerank_candidate_limit` | 10 | 候选太多 |
| 答案片段不够 | `max_sections` | 8 | 片段数太少 |
| 历史丢失 | `MAX_HISTORY_ROUNDS` | 10 | 历史轮数太少 |
| 生成答案太死板 | `temperature`（生成阶段） | 0.3 | 温度太低 |
| 召回太慢 | `max_events` | 50 | 粗排候选太多 |

### 9.2 按场景预设

| 场景 | 推荐配置 |
|------|---------|
| **快速问答** | `max_hops=1`, `rerank_candidate_limit=10`, `max_sections=3` |
| **深度问答** | `max_hops=3`, `rerank_candidate_limit=30`, `max_sections=8` |
| **精准问答** | `similarity_threshold=0.5`, `max_hops=1`, `rerank_top_k=3` |
| **多轮对话** | `MAX_HISTORY_ROUNDS=10`, `temperature=0.0` |
| **创意问答** | `temperature=0.5`, `max_sections=8` |

## 十一、关键设计决策

### 11.1 为什么 `temperature=0.0`？

[deepseek.py:26](file:///d:/ai-llamaindex/ai_sag/llm/deepseek.py#L26)

| 环节 | 是否需要确定性 | 原因 |
|------|--------------|------|
| 查询重写 | ✅ 是 | 相同历史应产生相同 eff_query |
| 实体抽取 | ✅ 是 | 相同 query 应抽相同实体 |
| 精排 | ✅ 是 | 相同候选应产生相同排序 |
| 答案生成 | ⚠️ 可调 | 创意场景可调高，但默认 0 保证可复现 |

### 11.2 为什么 `MAX_HISTORY_ROUNDS=5`？

[qa_engine.py:28](file:///d:/ai-llamaindex/ai_sag/retrieval/qa_engine.py#L28)

- 5 轮（10 条消息）足以覆盖大多数多轮对话场景
- 过多历史会导致 prompt 超长，超过 LLM context window
- DeepSeek-chat 默认 32K context，5 轮 + 检索片段约 4-8K token，安全

### 11.3 为什么 `max_hops=2`（而非论文的 1）？

[config.py:107](file:///d:/ai-llamaindex/ai_sag/base/config.py#L107)

- 信贷系统文档跨文档实体关联密集
- 1 跳可能漏召回（问 A 答 B 找不到）
- 2 跳增强召回能力，后续粗排 + 精排过滤噪声

### 11.4 为什么片段编号每轮重新开始？

[qa_engine.py:145-153](file:///d:/ai-llamaindex/ai_sag/retrieval/qa_engine.py#L145-L153)

- 简单直观，LLM 易引用
- 前端 `_lastSections` 缓存当前轮，hover 索引简单
- 多轮场景下历史【片段N】被清洗为"（见上文）"（方案 A 修复）
