# ai_sag 用户问答处理流程

本文档梳理 ai_sag 知识库问答系统从用户请求到答案返回的完整处理链路，包含**概要流程**、**流程细节**、**实现逻辑**三个层次。

---

## 一、概要流程

整体架构遵循 SAG（Semantic Association Graph）论文思路：**实体 + 事件双路召回 → 图扩展 → 粗排 → LLM 重排 → 向量补充 → LLM 生成**。

```
用户 Query
   │
   ▼
┌─────────────────────────────┐
│  0. 查询重写（多轮对话时）    │  ← 结合历史消解代词，得到独立可检索的 eff_query
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  1. Query Embedding         │  ← embedder.aembed_text(eff_query)
└─────────────┬───────────────┘
              │
       ┌──────┴──────┐ 并发
       ▼             ▼
┌─────────────┐ ┌──────────────────┐
│ 2a. 实体抽取 │ │ 2b. 事件标题向量召回 │
│  (LLM)      │ │  (event_titles)   │
└──────┬──────┘ └─────────┬────────┘
       │                  │
       ▼                  │
┌─────────────┐            │
│ 3. 实体召回   │            │
│ (精确+向量)  │            │
└──────┬──────┘            │
       │                  │
       ▼                  ▼
┌─────────────────────────────┐
│  4. 合并事件池 (seed_ids)     │  ← 去重合并实体事件集 + 标题事件集
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  5. 多跳 BFS 图扩展           │  ← 通过实体-事件关联扩展 max_hops 跳
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  6. 粗排 (余弦相似度)         │  ← 用 event_contents 向量与 query_vec 计算余弦
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  7. LLM 精排（融入实体角色）   │  ← 前 N 个候选交给 LLM 重排，角色信息作为重要信号
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  8. 切片回取 (Chunk A)        │  ← 根据重排后的事件 ID 回取 chunk 原文
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  9. 双路融合 (Chunk B 补充)   │  ← supplement/concat 模式，向量召回补足语义邻近片段
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│ 10. LLM 答案生成              │  ← achat / astream_chat，引用【片段N】
└─────────────┬───────────────┘
              │
              ▼
         返回前端
```

**核心入口**：

| 接口 | 路径 | 用途 | 引擎方法 |
|------|------|------|----------|
| 检索 | `POST /api/search` | 仅检索不回答 | `QAEngine.search` |
| 单轮问答 | `POST /api/ask` | 一问一答 | `QAEngine.ask` |
| 多轮对话 | `POST /api/chat` | 带历史的完整回答 | `QAEngine.chat` |
| 流式对话 | `POST /api/chat/stream` | SSE 流式输出 | `QAEngine.chat_stream` |

## 二、流程细节

### 阶段 0：查询重写（Query Rewrite）

- **触发条件**：仅多轮对话（`history` 非空），单轮问答跳过。
- **作用**：消解代词（"它/这家公司/那个产品"）和省略主语，把依赖上文的追问重写成一个独立、完整、可直接检索的查询。
- **实现**：
  - 取最近 `_REWRITE_MAX_ROUNDS=5` 轮历史，格式化为 `用户：xxx\n助手：xxx` 文本；
  - 调用 LLM（`achat`），使用 `QUERY_REWRITE_SYSTEM_PROMPT` + `QUERY_REWRITE_USER_PROMPT`；
  - 清洗返回：去掉引号、去掉"重写后："等前缀；若失败直接回退原 query。
- **产物**：`eff_query = rewritten or query`，后续所有步骤都使用 `eff_query`。

代码位置：[sag_retriever.py:133-170](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L133-L170)

### 阶段 1：向量编码

- 对 `eff_query` 调用 `embedder.aembed_text()` 生成 `query_vec`（嵌入向量）。
- 与下一步**并发执行**（`asyncio.gather`），减少时延。

### 阶段 2a / 2b：并发双路召回（无依赖）

**2a. 查询实体抽取**（LLM）：
- 使用 `astructured_predict` 让 LLM 输出 `_QueryEntities { entities: list[str] }`；
- Prompt：`query_extract_system_prompt()` + `QUERY_EXTRACT_USER_PROMPT`；
- 覆盖 11 类实体（time/location/person/organization/group/topic/work/product/action/metric/label），使用与入库完全一致的实体规范和命名归一化规则（去"有限公司"后缀、去地名修饰等）；
- 失败时降级：`_regex_entities()` 当前返回空列表（容错）。

**2b. 事件标题向量召回**：
- 在 `event_titles` 集合上做 ANN 检索：`vectors.aquery_event_titles(query_vec, top_k=max_events, similarity_threshold, source_ids)`；
- 得到 `title_event_ids`（按相似度降序）。

代码位置：[sag_retriever.py:72-83](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L72-L83)

### 阶段 3：实体召回（Entity Recall）

依赖阶段 2a 的 `query_entities`，分两步：

1. **精确匹配**：`db.search_entities_by_name(names, source_ids)` → `exact_ids`；
2. **向量模糊匹配**（对每个实体名）：
   - 批量编码：`embedder.aembed_texts(names)`；
   - **并发**对每个实体名向量在 `entities` 集合上 ANN 检索（top_k=5，阈值 `entity_expand_threshold=0.9`）；
   - 阈值独立且更严格（0.9 vs 事件 0.4），防止语义漂移引入噪声实体；
3. 合并去重 → `candidate_ids`，再按 `source_ids` 过滤；
4. 反查事件：`db.get_event_ids_by_entity_ids(entity_ids, source_ids)` → `entity_event_ids`。

代码位置：[sag_retriever.py:187-207](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L187-L207)

### 阶段 4：合并事件池（Seed Merge）

```python
seed_ids = list(dict.fromkeys(entity_event_ids + title_event_ids))  # 保序去重
```

- 若 `seed_ids` 为空，直接走**兜底路径**（fallback）：纯 chunk 向量召回 `_fallback()`，不进行图扩展/重排。
- `trace.seed_event_ids` 记录。

### 阶段 5：多跳 BFS 图扩展

- 目标：从种子事件出发，沿着"事件 ↔ 实体 ↔ 事件"的关联边做多跳扩展，召回与查询实体语义关联的周边事件（解决"问 A 但答案在 B"的跨事件推理问题）。
- 算法：
  - 维护 `tracked_events`（已跟踪事件集）和 `tracked_entities`（已触达实体集）；
  - 每一跳：取当前层事件 → 取出它们的所有实体 → 过滤掉已跟踪实体 → 用新实体反查事件（排除已跟踪事件）→ 加入下一层；
  - 跳数上限 `max_hops=2`（可配置），无新实体或无新事件时提前终止。
- 产物：`expanded_ids`（种子 + 扩展的全部事件），记录到 `trace.expanded_event_ids`。

代码位置：[sag_retriever.py:209-231](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L209-L231)

### 阶段 6：粗排（Coarse Rank）

- **为什么要粗排**：多跳扩展后事件可能达数百个，全部送入 LLM 精排不现实，先用向量相似度做一次廉价筛选。
- 实现：
  - 批量取出事件内容向量：`vectors.aget_embeddings("event_contents", event_ids)`；
  - 用 numpy 计算每个事件内容向量与 `query_vec` 的**余弦相似度**；
  - 按相似度降序排列，取前 `max_events=100` 个 → `top_n`。

代码位置：[sag_retriever.py:233-258](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L233-L258)

### 阶段 7：LLM 精排（含实体角色）

- 输入：`rerank_input = top_n[:rerank_candidate_limit=100]`（控制 prompt 长度）；
- 关键增强：**将查询实体在每个事件中的"角色"（role）信息注入候选描述**：
  - 查询阶段：`db.get_entity_names_by_ids(query_entity_ids)` 取实体名；
  - 对每个事件，遍历其 `entity_roles`（入库时抽取的实体-角色映射），若实体命中查询实体集，则渲染为 `"华为(贷款方)"`、"2024年(签约时间)"等；
  - 额外标注命中数：`[命中 2/3]`；
- 让 LLM 用 `astructured_predict` 输出 `_RerankOrder { order: list[int] }`（候选编号，按相关性降序）；
- 若结构化输出失败，降级为 `acomplete` + 正则提取数字；
- 结果处理：LLM 返回的顺序优先，未被 LLM 提及的候选按原序追加在末尾（保证不丢）；
- 最终取前 `rerank_top_k=5` 个事件 ID。

代码位置：[sag_retriever.py:260-328](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L260-L328)

### 阶段 8：切片回取（Chunk A）

- 把精排后的事件 ID 映射回原文 chunk：
  - `db.get_chunk_ids_by_event_ids(reranked_ids)` → `ev_chunk_map`；
  - `db.get_chunks_by_ids(chunk_ids)` → 取 chunk 内容；
- 每个 chunk 包装为 `RetrievedSection(chunk_id, source_id, document_id, heading, content, rank, score)`；
- 顺序严格遵循 `reranked_ids`（精排顺序即最终顺序），`rank` 按位置递增、`score` 线性衰减（1.0, 0.99, 0.98, ...）。

代码位置：[sag_retriever.py:330-353](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L330-L353)

### 阶段 9：双路融合（Fusion）

事件图谱路径（Chunk A）擅长**关系型/跨段推理**，纯向量路径（Chunk B）擅长**语义邻近但事件图未覆盖**的兜底，二者互补，支持两种融合模式：

| 模式 | 行为 |
|------|------|
| `supplement`（补充，默认） | Chunk A 不足 `max_sections` 时，用 chunk 向量召回补足剩余配额 |
| `concat`（拼接） | A、B 各占一半配额（A 优先），合并去重 |

- Chunk B 实现：`vectors.aquery_chunks(query_vec, top_k=2*max_sections, ...)` → 按 `source_ids` 过滤 → 跳过已存在 chunk；
- 去重使用 `chunk_id` 集合；
- 最终截断为 `max_sections=5` 条。

代码位置：[sag_retriever.py:357-422](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L357-L422)

### 阶段 10：答案生成（Generation）

检索完成后由 `QAEngine` 负责组装 prompt 并调用 LLM：

**单轮问答（ask）**：
- `_build_single_turn_messages`：`[SYSTEM(含资料), USER(query)]`
- 调用 `llm.achat(messages)` 返回完整答案字符串。

**多轮对话（chat）**：
- `_build_messages`：`[SYSTEM(含资料), ...历史 user/assistant（截断至最近5轮）, USER(query)]`
- 历史对话**不重复携带资料**，只保留问答文本，避免 token 膨胀；
- 片段编号采用当前检索结果的全局递增编号，避免多轮引用错乱。

**流式对话（chat_stream）**：
- 先同步返回检索结果（`meta` 事件，含 sections/trace），前端用于渲染引用标签；
- 然后通过 `llm.astream_chat(messages)` 异步迭代 token，以 `delta` 事件 SSE 推流；
- 最后发 `done` 事件结束，异常时发 `error` 事件。

**上下文构建**：
- 每个片段格式：`【片段{i}】(来源: {source_short}, 小节: {heading})\n{content}`；
- System Prompt 要求：仅依据资料作答、不杜撰、引用片段编号、结合历史消解代词。
- 若 `sections` 为空，直接返回 `QA_EMPTY_ANSWER = "未检索到相关内容，无法回答该问题。"`

## 三、实现逻辑要点

### 3.1 核心设计决策

1. **全链路异步**
   - MySQL 使用 `aiomysql`（异步驱动 + 连接池）；
   - LLM 使用 LlamaIndex 原生异步方法（`achat`/`astream_chat`/`astructured_predict`）；
   - Embedder/VectorStore 为 CPU 密集同步组件，用 `asyncio.to_thread` 包装为 `aembed_text*` / `aquery_*` 异步接口；
   - 无依赖步骤全部 `asyncio.gather` 并发（如实体抽取 + 标题召回、多个实体名并发向量查询）。

2. **入库与查询口径一致**
   - 11 类实体类型、实体命名规范（去"有限公司"等）在入库抽取和查询抽取中**共用同一份常量**（`prompts.py` 中 `ENTITY_TYPES`、`ENTITY_NAME_NORM`），避免口径漂移导致命中率下降。

3. **实体角色（role）信号融入精排**
   - 入库阶段为每个 (事件, 实体) 对抽取角色标签（如"贷款方/担保方/签约时间/标的物"）；
   - 精排时把查询实体在候选事件中的角色渲染给 LLM（如"众商贷(标的物)"），使 LLM 能区分"A 向 B 贷款"与"B 向 A 贷款"等方向性问题，显著提升关系型查询准确率。

4. **双路融合兜底**
   - 图谱路径擅长结构化关系推理，纯向量路径擅长语义相似兜底；
   - 默认 `supplement` 模式保证图谱优先，`concat` 模式用于需要更广覆盖的场景；
   - seed_ids 为空时直接回退到纯向量召回（`_fallback`），避免"查不到就完全没结果"。

5. **多层截断与保护**
   - `rerank_candidate_limit=100`：限制送入 LLM 精排的候选数，防止 prompt 过长；
   - `rerank_top_k=5`：最终精排输出事件数，对应 `max_sections=5` 个片段；
   - `MAX_HISTORY_ROUNDS=5`：多轮历史截断轮数，防止上下文爆炸；
   - `entity_expand_threshold=0.9`：实体模糊匹配阈值严格，抑制噪声实体。

6. **容错降级**
   - 查询重写失败 → 用原 query；
   - LLM 实体抽取失败 → 空列表（regex 兜底）；
   - LLM 结构化精排失败 → 正则解析 + 原序补齐；
   - 精排整体失败 → 直接用粗排 top_k；
   - 检索无结果 → 返回固定兜底文案；
   - 答案生成异常 → 返回错误消息字符串而非抛异常。

### 3.2 关键算法

**余弦相似度**（粗排，numpy 实现）：
```python
score = dot(q, e) / (||q|| * ||e|| + 1e-8)
```
加入 `1e-8` 防止零向量除零；缺失 embedding 的事件得分 0.0。

代码位置：[sag_retriever.py:241-258](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L241-L258)

**多跳 BFS 扩展**：维护 `tracked_events` / `tracked_entities` 两个集合避免环和重复访问；每跳只处理"新发现"实体带来的新事件，保证扩展方向向外扩散。

**保序去重**：`list(dict.fromkeys(...))` — 利用 Python 3.7+ dict 插入有序特性，既去重又保留靠前路径的优先级（实体路径优先于标题路径）。

### 3.3 异常处理与可观测性

- 每个接口入口打 INFO 日志（query、命中 chunk 数、关键长度）；
- 异常打 EXCEPTION 日志带完整栈，API 层统一封装为 HTTP 500；
- `SearchTrace` 贯穿全链路，记录 `query_entities / seed_event_ids / expanded_event_ids / reranked_ids / fallback`，便于排查"为什么召回/没召回"；
- 流式接口异常通过 `error` SSE 事件回传前端，不打断连接。

## 四、模块调用关系

```
api.py (FastAPI 路由)
  └─> QAEngine (qa_engine.py)           # 编排层：search/ask/chat/chat_stream
        ├─> SagRetriever (sag_retriever.py)  # 检索核心
        │     ├─> BaseEmbedder               # 文本向量化
        │     ├─> BaseVectorStore            # Chroma 向量 ANN 检索
        │     │     └─ collections: chunks / event_titles / event_contents / entities
        │     ├─> MysqlStore                 # aiomysql 异步存储
        │     │     ├─> sources/documents

        │     │     ├─> events / event_entities(含 role)

        │     └─> LLM (llm/)                # DeepSeek/OpenAI 等
        │           ├─> achat / acomplete

        │           └─> astructured_predict
        └─> LLM (生成答案/流式输出)
```
