## 四、隐患详细分析与解决方案

### 隐患 1：`deleted_at` 字段无独立索引

**代码位置**：[schema.sql:47](file:///d:/ai-llamaindex/ai_sag/storage/schema.sql#L47)

```sql
deleted_at  DATETIME NULL,
```

**根因**：
- 软删除字段 `deleted_at` 没有独立索引
- 查询①②都需过滤 `deleted_at IS NULL`
- MySQL 在 JOIN 后需对匹配行逐行扫描过滤

**影响**：
- 查询①：因 `id IN` 已走主键，影响较小
- 查询②：JOIN 后过滤，数据量大时（> 10万事件）扫描行数显著增加

**严重程度**：★★ 中

**触发阈值**：`aisag_events` 表活跃事件数 > 5万

**解决方案**：新增 `deleted_at` 索引

```sql
ALTER TABLE aisag_events
ADD INDEX idx_aisag_event_deleted (deleted_at);
```

**验证 SQL**：

```sql
EXPLAIN SELECT DISTINCT ee.event_id
FROM aisag_event_entities ee
JOIN aisag_events e ON e.id = ee.event_id
WHERE ee.entity_id IN ('实体UUID1')
  AND e.deleted_at IS NULL;
```

期望执行计划：

| 表 | type | key | rows |
|----|------|-----|------|
| ee | range | `idx_aisag_ee_entity` | 少量 |
| e | eq_ref | `PRIMARY` | 1 |

### 隐患 2：`NOT IN` 排除效率有限

**代码位置**：[mysql_store.py:301](file:///d:/ai-llamaindex/ai_sag/storage/mysql_store.py#L301)

```python
ph_x = ",".join(["%s"] * len(exclude))
where.append(f"ee.event_id NOT IN ({ph_x})")
params.extend(exclude)
```

**根因**：
- MySQL 对 `NOT IN` 无法高效利用索引排除
- 多跳累积 `tracked_events` 后，每次都要对结果集做线性过滤

**影响**：
- `tracked_events` 累积到几十、几百个时，过滤成本上升
- 因 `entity_id IN` 已先收敛结果集，影响有限

**严重程度**：★ 低

**触发阈值**：`tracked_events` 集合 > 50 个

**解决方案**：改用 `LEFT JOIN ... IS NULL`

```python
# 改前
sql = (f"SELECT DISTINCT ee.event_id FROM aisag_event_entities ee "
       f"JOIN aisag_events e ON e.id = ee.event_id WHERE " + " AND ".join(where))

# 改后（用临时表 + LEFT JOIN 排除）
if exclude:
    ph_x = ",".join(["%s"] * len(exclude))
    sql = (f"SELECT DISTINCT ee.event_id FROM aisag_event_entities ee "
           f"JOIN aisag_events e ON e.id = ee.event_id "
           f"LEFT JOIN (SELECT id FROM aisag_events WHERE id IN ({ph_x})) ex "
           f"ON ee.event_id = ex.id WHERE ex.id IS NULL AND " + " AND ".join(other_where))
```

### 隐患 3：`SELECT DISTINCT` 临时表去重

**代码位置**：[mysql_store.py:303](file:///d:/ai-llamaindex/ai_sag/storage/mysql_store.py#L303)

```sql
SELECT DISTINCT ee.event_id FROM aisag_event_entities ee ...
```

**根因**：
- 一个实体可能关联多个事件，`DISTINCT` 需去重
- MySQL 会使用临时表做去重操作

**影响**：
- 结果集大时（> 1000 行）消耗内存与排序成本

**严重程度**：★ 低

**触发阈值**：单跳扩展结果 > 1000 行

**解决方案**：改用 `GROUP BY` 或应用层去重

```sql
-- 方案A：GROUP BY
SELECT ee.event_id FROM aisag_event_entities ee
JOIN aisag_events e ON e.id = ee.event_id
WHERE ee.entity_id IN (...)
GROUP BY ee.event_id;

-- 方案B：应用层去重（Python set 去重，更可控）
rows = await self._fetchall(sql_without_distinct, params)
return list({r["event_id"] for r in rows})
```

### 隐患 4：JOIN 放大效应

**代码位置**：[mysql_store.py:304](file:///d:/ai-llamaindex/ai_sag/storage/mysql_store.py#L304)

```sql
JOIN aisag_events e ON e.id = ee.event_id
```

**根因**：
- 实体 → 事件 → 实体 多对多关系，每跳 JOIN 放大结果集
- `max_hops=2` 时扩展事件数可能指数增长

**影响**：
- 多跳后事件数从 5 个涨到 50+ 个
- 后续粗排、精排都受影响

**严重程度**：★★ 中

**触发阈值**：扩展事件数 > 200

**解决方案**：每跳后即时裁剪 TopK

```python
# [sag_retriever.py:215-228] 改造
async def _expand(self, seed_ids, initial_entity_ids, source_ids):
    tracked_events = set(seed_ids)
    tracked_entities = set(initial_entity_ids)
    current_event_ids = list(seed_ids)

    MAX_EVENTS_PER_HOP = 50   # 新增：每跳事件上限

    for _ in range(self.cfg.search.max_hops):
        if not current_event_ids:
            break
        events = await self.db.get_events_by_ids(current_event_ids, source_ids)
        # ... 现有逻辑
        new_event_ids = await self.db.get_event_ids_by_entity_ids(
            new_entities, source_ids, exclude=list(tracked_events))
        tracked_events.update(new_event_ids)
        current_event_ids = new_event_ids[:MAX_EVENTS_PER_HOP]  # 裁剪

    return list(tracked_events)
```

### 隐患 6：`entity_ids` 全量加载到内存

**代码位置**：[sag_retriever.py:220-221](file:///d:/ai-llamaindex/ai_sag/retrieval/sag_retriever.py#L220-L221)

```python
events = await self.db.get_events_by_ids(current_event_ids, source_ids)
current_entity_ids = set()
for ev in events:
    current_entity_ids.update(ev.entity_ids)   # 全量加载
```

**根因**：
- 每跳加载所有事件的所有实体到内存
- 大事件集内存占用高

**影响**：
- 单跳事件 > 100 个时内存波动明显

**严重程度**：★ 低

**触发阈值**：单跳事件数 > 100

**解决方案**：流式处理 + 增量去重

```python
# 流式处理，避免一次性 set 合并
new_entities = []
for ev in events:
    for eid in ev.entity_ids:
        if eid not in tracked_entities:
            tracked_entities.add(eid)
            new_entities.append(eid)
    # 达到上限提前退出
    if len(new_entities) > 200:
        break
```

## 五、优先级与处理建议

### 优先级矩阵

| 优先级 | 隐患# | 处理时机 | 理由 |
|--------|-------|---------|------|
| **P0 立即处理** | — | — | 当前数据规模下均无感，无需处理 |
| **P1 关注监控** | 1, 4, 7, 9 | 数据量增长后 | 影响显现，建议加监控告警 |
| **P2 长期优化** | 2, 3, 5, 6, 8, 10 | 按需优化 | 收益有限，可延后 |

### 触发阈值参考

| 隐患# | 安全范围 | 警戒范围 | 危险范围 | 触发动作 |
|-------|---------|---------|---------|---------|

| 事件主键查询 | ✅ 走 PRIMARY KEY |
| `deleted_at` 过滤 | ⚠️ 无索引，数据量大时显现 |
| JOIN 放大 | ⚠️ 需配合每跳裁剪 |
| 当前数据规模 | ✅ 性能无感知影响 |
| 优化时机 | 📌 事件数突破 5万 或 单轮扩展 > 200 时启动 |

**当前建议**：所有隐患在现有数据规模下都不影响性能，无需立即修改。待事件数突破 **5万** 或单轮多跳扩展事件数突破 **200** 时，再启动隐患 1、4、7 的优化。
